# Feature Specification: Branded UI Polish

**Feature Branch**: `branded-ui-polish`

**Created**: 2026-05-17

**Status**: Draft

**Input**: User description: "Branded UI polish — a coordinated visual identity pass across the marketing surface, the authenticated shell, the accounts feature, and every empty state. Replace the generic shadcn appearance with a distinct Abacus identity (custom abacus brand mark, favicon, OG image, Inter type, money-positive accent, money rendering primitive with tabular numerals, dashboard footer, sidebar grouping, richer empty states with previews, dashboard home rewrite, marketing polish). A chore — no new product capability; no roadmap number consumed."

## Why

Abacus today is competent and accessible but visually generic — it looks like a default shadcn/ui project. Personal-finance apps live or die on perceived trust and warmth, and "generic engineer's project" actively undermines that. The product surface is still small (one real feature page — Accounts — and four placeholder routes), so this is the cheapest possible moment to bake a distinct visual identity in: every future feature (Categories, Transactions, Budgets, Dashboard widgets, Reports) will inherit a polished foundation instead of bolting onto a generic shell that has to be retrofitted later. The work is cross-cutting (it touches the marketing surface, the authenticated shell, the accounts list, and every empty state) but introduces zero new product capability and no new domain entities, so it lands as a chore rather than a numbered roadmap feature. Roadmap numbers (005 Categories, 006 Transactions, 007 Real dashboard, 008 Budgets, ...) are unchanged.

## Clarifications

### Session 2026-05-17

- Q: Should positive monetary amounts in the money rendering primitive use the new money-positive accent color, or render in the default foreground color, with only the money-positive accent reserved for explicitly positive UI affordances (e.g., income labels in feature 007)? → A: Foreground default for positive amounts. The `money-positive` accent green is RESERVED for explicit income / gain semantics that future features (007 Dashboard income widget, 015 Charts series, 008 Budget surplus, etc.) introduce. Negative amounts still use desaturated red (FR-008). Zero amounts still use muted-foreground (FR-013). Rationale: in a personal-finance app, "positive balance" is the default state of the data — painting every positive amount green dilutes the signal and consumes affordance-color-space that future features need for "this is income vs that is the running balance." Matches the visual conventions of Copilot Money, Monarch, and Mercury.
- Q: For the not-yet-shipped placeholder routes (`/dashboard/transactions`, `/dashboard/budgets`), should the empty-state copy explicitly name the roadmap feature that will fill the route (e.g., "shipping with feature 006") and link to a public roadmap, or stay deliberately commitment-free with a generic "coming soon" message and no external link? → A: Descriptive but not date-committal. Each placeholder empty state names what the feature WILL DO in one short line (the *concept*, not the *timeline*) — e.g., "Track every dollar in and out of your accounts, categorise them, and see where your money goes" for Transactions; "Cap your spending by category and stay on top of the limits you set" for Budgets. No feature number references, no `/roadmap` route, no external links. The internal `docs/roadmap.md` stays internal. Ages gracefully whether the feature ships in a week or a quarter. Rationale: option A (feature number + public roadmap) requires adding a route and commits to a release cadence we don't have; option B (generic "coming soon") leaves the user no information; option C balances information with non-commitment.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A new visitor instantly recognises Abacus as a deliberate personal-finance product (Priority: P1)

As someone who has just received a link to Abacus and lands on the marketing surface for the first time, I see a custom abacus brand mark in the header, a tailored sans-serif typeface, a clear product tagline, and a coherent violet-tinted brand palette. When I share the URL with a colleague in Slack or iMessage, the preview shows a recognisable abacus mark + the Abacus wordmark + the tagline on a branded background — not a generic Next.js default. The browser tab shows an abacus-shaped favicon at typical favicon sizes.

**Why this priority**: First impressions on a personal-finance app are load-bearing — generic visuals erode trust before the product has had a chance to demonstrate correctness. This story is the single biggest perceived-quality lever this chore delivers, and it is the only story whose value is mostly captured at the marketing surface.

**Independent Test**: Open `/` in a fresh browser session. Confirm the header renders the custom abacus brand mark (not a generic wallet/coin glyph), confirm the browser tab favicon renders as a recognisable abacus, confirm the page font has visibly switched to the chosen brand typeface, and confirm the brand wordmark "Abacus" appears alongside the mark. Paste the URL into any tool that resolves Open Graph metadata (Slack message composer, Twitter card validator, iMessage preview, or any equivalent) and confirm the resolved preview shows the abacus mark + the wordmark + the tagline on a branded background.

**Acceptance Scenarios**:

1. **Given** a first-time visitor with no Abacus cookies, **When** they load `/`, **Then** the marketing header renders the custom Abacus brand mark in place of any generic icon, the wordmark "Abacus" sits next to the mark, the brand typeface is in effect (not the browser default), and the page paints without a noticeable swap from a fallback font.
2. **Given** any browser that supports favicons, **When** the visitor looks at the browser tab for `/`, **Then** the favicon renders as a recognisable abacus mark at favicon size (frame + at least two visible rods + at least four visible beads) and uses the brand accent color on the beads.
3. **Given** any third-party tool that resolves Open Graph metadata for `https://<host>/`, **When** the tool fetches the page's social preview, **Then** the preview shows the abacus brand mark, the wordmark "Abacus", and the product tagline on a branded background, sized and laid out for typical social-share previews.
4. **Given** the marketing hero, feature grid, changelog, and footer all rendered, **When** the visitor scrolls the marketing page, **Then** every section uses the same brand typeface, the same brand mark consistently, and the same accent color discipline (no leftover generic shadcn pop colors).

### User Story 2 — An authenticated user sees one coherent brand across every dashboard route (Priority: P1)

As a signed-in user navigating between `/dashboard`, `/dashboard/accounts`, `/dashboard/transactions`, `/dashboard/budgets`, and `/dashboard/settings`, I see the same custom abacus brand mark in the sidebar, the same brand typeface across every page, a footer that appears consistently at the bottom of every authenticated route (parity with the marketing footer), and a sidebar nav organised into named groups that telegraphs how the app is structured. The dashboard home no longer shows a stale "feature not available" placeholder — it shows a welcoming panel that correctly reflects what the app does today (Accounts is shipped; Transactions and Budgets are upcoming).

**Why this priority**: Once a user is past the marketing surface, they spend 100% of their time inside the authenticated shell. Any inconsistency between routes inside the shell — a missing footer here, a stale "future feature" caption there, a generic icon in the sidebar — reads as the product not being finished. This story is the second leg of the first-impression bet from US1 and is the bulk of the day-to-day perceived-quality return.

**Independent Test**: Sign in as any user, then visit `/dashboard`, `/dashboard/accounts`, `/dashboard/transactions`, `/dashboard/budgets`, and `/dashboard/settings` in turn. Confirm at every route: (a) the sidebar shows the custom Abacus brand mark and wordmark, (b) the brand typeface is in effect on the page body and headings, (c) a footer is rendered at the bottom of the viewport (sticky on short pages, scrolling on long ones), and (d) the sidebar nav is grouped into the documented sections rather than a single flat list. On `/dashboard` specifically, confirm the content is a welcome panel that links to Accounts and is no longer a stale "future feature" placeholder.

**Acceptance Scenarios**:

1. **Given** a signed-in user on any authenticated route, **When** they look at the sidebar, **Then** the brand area shows the custom abacus mark and the "Abacus" wordmark, both using the brand typeface, and the nav items are grouped into the documented sections with section labels rendered in the agreed tertiary-label treatment.
2. **Given** a signed-in user on any authenticated route, **When** they scroll to the bottom of the viewport on a short page, **Then** a footer is pinned at the bottom of the viewport. On a long page, the footer sits at the natural end of the page content after scrolling. The footer is present and visually consistent on every authenticated route.
3. **Given** a signed-in user navigates from one authenticated route to another, **When** the new route renders, **Then** the brand mark, the brand typeface, the sidebar grouping, and the footer all remain in place without any visible inconsistency between routes.
4. **Given** a signed-in user with at least one account, **When** they visit `/dashboard`, **Then** the page renders a welcome panel using the upgraded empty-state primitive, including the abacus mark as the panel's illustration, copy that names what is available today (Accounts) and what is coming (Transactions, Budgets), and a primary call-to-action that links to `/dashboard/accounts`. The legacy "future feature" caption and the disabled "Add your first account" affordance MUST be gone.
5. **Given** a signed-in user on any authenticated route, **When** they look for the theme toggle, **Then** the toggle is reachable from every authenticated route in the same location (the exact location — header vs footer — is plan-level).

### User Story 3 — A user reading the accounts list trusts the money rendering at a glance (Priority: P1)

As a user looking at the accounts list with multiple accounts (potentially in different currencies), I can read the balance column at a glance: digits align vertically in a column because the typography uses tabular numerals; the balance column is right-aligned so amounts compare visually; positive, zero, and negative amounts are visually distinguished by color in a way that is calibrated for a money UI (not alarming, not arbitrary); and every balance is rendered together with its currency, never as a bare number. The balance column is the most visually prominent column on the page.

**Why this priority**: This is the highest-leverage rendering change in the chore. Money rendering is the moment where a personal-finance app most directly earns or loses trust. The plumbing for money values already exists (constitution Principle I, the `lib/money/` module landed in feature 004); this story upgrades only the display layer so the existing values *look* trustworthy as well as being internally correct.

**Independent Test**: As a user with at least three accounts (mix of positive balances, zero balance, and at least one negative balance on a `CREDIT` account, ideally in two different currencies), open `/dashboard/accounts`. Confirm: (a) the balance column is right-aligned; (b) digits in different rows line up vertically (tabular numerals); (c) the positive, zero, and negative rows are visually distinguished from each other; (d) every balance shows its currency alongside the amount, with no bare numbers anywhere; (e) the balance column is the most prominent column visually (typography weight, color contrast, or both).

**Acceptance Scenarios**:

1. **Given** a user with one or more accounts, **When** they view `/dashboard/accounts`, **Then** every balance value is rendered with tabular numerals so that digits in different rows align vertically in the column.
2. **Given** the accounts list is rendered, **When** the user looks at the balance column, **Then** the column is right-aligned and the balance value is the most prominent piece of typography in the row (heavier weight, larger size, or stronger color than the other columns).
3. **Given** a user has accounts with a positive balance, a zero balance, and a negative balance, **When** they view the list, **Then** the three sign states are visually distinguishable from each other through color: positive amounts render in the default `foreground` color, zero amounts render in `muted-foreground`, and negative amounts render in the desaturated red derived from `--destructive`. The new `money-positive` green is NOT applied to positive amounts by this chore (it is reserved for explicit income/gain semantics in future features). Color MUST NOT be the sole carrier of sign information — the sign character itself remains visible.
4. **Given** any account row is rendered, **When** the balance is shown, **Then** the value appears together with its currency (e.g., `$1,250.00` with the currency mark next to the amount, or `1,250.00 USD` with the currency code adjacent — the exact composition is plan-level). The system MUST NOT render a bare amount without its currency anywhere in this list (constitution Principle I, reaffirmed via FR-011 in feature 004).
5. **Given** a user opens the create or edit side sheet for an account, **When** the form renders the current balance as a read-back, **Then** the same money-rendering treatment from the list (tabular numerals, currency adjacent, sign-aware color) is in effect on the read-back, even if the editable input itself uses a plain numeric input control.

### User Story 4 — A user landing on a not-yet-shipped route sees a rich, calibrated empty state (Priority: P2)

As a user who clicks into `/dashboard/transactions` or `/dashboard/budgets` before those features have shipped, I see an empty state that is genuinely informative — a brand-consistent illustration, a clear short headline, a description of what the page will do, and a faded "preview" mock of what the loaded experience will look like once data is present. The empty state on `/dashboard/settings` is simpler (no preview) because previewing form fields we haven't designed yet would mislead. The accounts page, when the user has zero accounts, uses the same upgraded primitive and looks deliberate rather than sparse.

**Why this priority**: Empty states are the first thing a user sees on three of five dashboard routes today. Sparse "nothing here yet" placeholders read as unfinished; rich previewed empty states read as "intentional, coming soon, here is what you'll get." This story does not block the brand-identity stories (US1, US2, US3) but compounds on top of them: it is the difference between "polished brand, sparse content" and "polished brand, deliberate content."

**Independent Test**: Visit each of `/dashboard`, `/dashboard/accounts` (with zero accounts), `/dashboard/transactions`, `/dashboard/budgets`, and `/dashboard/settings` in turn. Confirm at each route: (a) the empty state uses an illustration consistent with the brand (stroke-based, monochrome with one accent — NOT a stock illustration library and NOT the generic shadcn icon); (b) the copy is route-specific and accurate to what the product does today and will do later; (c) on `/dashboard/transactions` and `/dashboard/budgets`, a decorative non-interactive preview of the loaded state is visible below the call-to-action; (d) the preview is announced as decorative to assistive technology and is not part of the keyboard tab order; (e) on `/dashboard/settings`, no preview is rendered.

**Acceptance Scenarios**:

1. **Given** a user visits `/dashboard/transactions` before feature 006 has shipped, **When** the page renders, **Then** the upgraded empty-state primitive shows a brand-consistent illustration, a route-specific headline ("Transactions are coming soon" or equivalent), a one-line description of what the feature WILL DO (no roadmap number, no release-cadence commitment, no external link), and a decorative faded preview of a two-row transaction layout below the description. No primary call-to-action button is rendered.
2. **Given** a user visits `/dashboard/budgets` before feature 008 has shipped, **When** the page renders, **Then** the empty state uses the same primitive with a brand-consistent illustration, a route-specific headline ("Budgets are coming soon" or equivalent), a one-line description of what the feature WILL DO, and a decorative faded preview of a progress-bar widget below the description. The preview MUST be non-interactive and announced as decorative to assistive technology. No primary call-to-action button is rendered.
3. **Given** a user visits `/dashboard/settings` before feature 017 has shipped, **When** the page renders, **Then** the empty state uses the same primitive with a brand-consistent illustration, a route-specific headline, and a one-line description of what the feature WILL DO, but NO preview slot is rendered (rendering form-field mocks before the settings UX is designed would mislead users about what is coming). No primary call-to-action button is rendered.
4. **Given** a user visits `/dashboard/accounts` with zero accounts, **When** the page renders, **Then** the existing empty state from feature 004 is migrated onto the upgraded primitive with a brand-consistent illustration; the create-account call-to-action behavior is unchanged.
5. **Given** the upgraded empty-state primitive is used by a future feature that only has a single icon (not a full illustration), **When** that feature passes the legacy `icon` prop instead of the new illustration slot, **Then** the primitive MUST still render correctly (backward compatibility with the existing `EmptyState` shape established in feature 002).

### User Story 5 — A user navigating the sidebar can read the product structure at a glance (Priority: P2)

As a user looking at the sidebar — whether on desktop (fixed sidebar) or on mobile (drawer) — I see the navigation organised into named groups (one for routes that show what is happening with my money, another for routes that let me change settings or budgets) rather than a single flat list of five items. The active route is still clearly indicated, the keyboard tab order through the sidebar is still sensible, and the mobile drawer follows the same grouping so the structure does not change between viewports.

**Why this priority**: A two-group sidebar at five items reads as "this product has a deliberate structure" rather than "this product happens to have five things." It also makes the sidebar age well as features 005 (Categories), 006 (Transactions, already in the list), 008 (Budgets, already in the list), and 015 (Reports) land — each new route slots naturally into one of the groups instead of cluttering a growing flat list. P2 because the flat sidebar from feature 002 is functional today; this is a UX upgrade, not a fix.

**Independent Test**: Open the sidebar on desktop and confirm the nav items are organised into named groups with a section label above each group and a visible separator between groups. Resize to mobile width and open the drawer, confirm the same grouping appears inside the drawer. Tab through the sidebar with the keyboard, confirm focus moves through every nav item in a sensible order and that the active route is still highlighted. The exact group names and the assignment of each route to a group are documented in the functional requirements.

**Acceptance Scenarios**:

1. **Given** a desktop user (viewport ≥1024px), **When** they look at the sidebar, **Then** the nav items are organised into the documented groups, each group has a section label rendered in the agreed tertiary-label treatment, and a visible separator sits between groups.
2. **Given** a mobile user (viewport <640px) opens the drawer, **When** the drawer renders, **Then** the same grouping is visible inside the drawer with the same section labels and separator treatment as the desktop sidebar.
3. **Given** a user navigates to a route within any group, **When** the sidebar renders, **Then** the active route is visually distinguished from inactive routes (the active-state styling from feature 002 is preserved and is not regressed by the grouping change).
4. **Given** a keyboard-only user tabs through the sidebar, **When** they reach a section label, **Then** the section label MUST NOT receive focus (it is decorative); focus moves directly from one nav item to the next nav item, in the documented order.

### Edge Cases

- **A user disables JavaScript in their browser** — the marketing surface and the dashboard surface must still render meaningful, readable content (a server-rendered shell with the brand mark, the wordmark, the brand typeface fallback, and static text). The richer interactive affordances (theme toggle, mobile drawer) may degrade; the page MUST NOT render blank or broken.
- **The brand typeface fails to load** — the page falls back to the system sans-serif stack without any layout shift large enough to be perceptible as a content jump. The brand-typeface fallback is documented at the FR level so the planner picks a font-loading strategy that achieves this without an ambiguous trade-off.
- **The favicon must render on Safari pinned tabs** — Safari handles some favicon formats differently from other browsers; the favicon strategy MUST produce a recognisable abacus mark on Safari pinned tabs as well as in mainstream tab favicons.
- **The brand mark at 16px favicon size** — the abacus mark MUST remain recognisable at favicon size (frame + at least two visible rods + at least four visible beads visible to the eye). Designs that only read at 24px+ are unacceptable.
- **Social-share previews on multiple platforms** — the Open Graph image MUST render correctly on at least Slack, iMessage, and Twitter-style cards. Platform-specific custom previews (Twitter `summary_large_image`, Facebook custom dimensions, etc.) are NOT required, but the single OG image MUST be usable across these mainstream platforms without obvious cropping or unreadable text.
- **Dark mode contrast on the money-positive accent** — the money-positive accent color MUST meet WCAG AA contrast against both the light-mode background and the dark-mode background. A green that looks correct in light mode but fades into the dark-mode background is unacceptable.
- **Mobile drawer + dashboard footer interaction** — when the mobile drawer is open, the footer MUST NOT visually conflict with the drawer (e.g., footer content peeking through under a translucent drawer). The drawer takes visual precedence; the footer remains below the page content.
- **A user with zero active accounts AND zero archived accounts visits `/dashboard`** — the dashboard welcome panel still renders, still links to `/dashboard/accounts`, and is consistent with the empty state on the accounts page itself. The two empty states are different in purpose (one is a welcome, the other is a create-account prompt) but consistent in visual language.
- **A user with at least one account visits `/dashboard`** — the welcome panel is still appropriate (this chore intentionally keeps the dashboard home as a welcome until feature 007 ships real widgets). The copy MUST NOT imply "you have no accounts" when the user does have accounts.
- **The theme toggle is moved to the footer in plan** — the toggle MUST remain reachable on every authenticated route on every viewport width, including mobile (where the footer may sit below the fold). If the toggle is placed in the footer, the footer's discoverability on mobile MUST not regress access to theme switching.
- **An existing E2E Playwright test from feature 002, 003, or 004 walks an authenticated route** — every existing E2E test MUST continue to pass after the chore lands. This chore is a visual refresh, not a behavioral change.
- **A user encounters a route placeholder before the corresponding roadmap feature has shipped** — the empty state's call-to-action behavior is governed by the Clarifications session decision on coming-soon copy specificity (see Clarifications).

## Requirements *(mandatory)*

### Functional Requirements

#### Brand identity (visible everywhere)

- **FR-001**: The system MUST introduce a custom Abacus brand mark that visually depicts an abacus (frame + rods + beads). The brand mark MUST replace any generic icon currently used in either the marketing header brand area or the authenticated shell sidebar brand area. The brand mark MUST inherit the surrounding text color in both light and dark modes (the frame/strokes follow text color); only the bead fill MAY use the brand accent color (violet primary, established by feature 002).
- **FR-002**: The brand mark MUST be visually recognisable as an abacus at both the standard header size and at favicon size. "Recognisable at favicon size" means: at 16px, a user MUST be able to identify a frame, at least two distinct rods, and at least four distinct beads. Designs that read only at larger sizes are unacceptable.
- **FR-003**: The system MUST provide a favicon rendered from the same abacus brand mark via the project framework's standard favicon-generation convention. The favicon MUST render in mainstream desktop and mobile browser tabs and MUST be usable on Safari pinned tabs (where some SVG favicon strategies break).
- **FR-004**: The system MUST provide an Open Graph image for social sharing, rendered via the project framework's standard OG-image-generation convention. The OG image MUST show the abacus brand mark, the wordmark "Abacus", and the product tagline ("Personal finance, finally clear" or the established equivalent) on a brand-accent-tinted background. The image MUST render correctly when fetched by mainstream social-platform link previewers (Slack, iMessage, Twitter-style cards).
- **FR-005**: The system MUST adopt a brand sans-serif typeface (Inter or equivalent), loaded as a self-hosted web font via the project framework's standard font convention so that no third-party network request to a font CDN is required at runtime. The font MUST fall back gracefully to the system sans-serif stack if loading fails. The font-loading strategy MUST avoid a noticeable text-content-jump on initial paint (the exact display strategy — `swap` vs `optional` — is plan-level, but the perception bar is "no obvious flash of unstyled-then-styled content").
- **FR-006**: The wordmark "Abacus" MUST be rendered alongside the brand mark in both the marketing header brand area and the authenticated shell sidebar brand area. The brand mark and the wordmark together constitute the brand lockup; neither is rendered without the other in these two surfaces.
- **FR-007**: The system MUST introduce a `money-positive` brand-accent color (a soft, slightly desaturated green) as a CSS token, distinct from the `--primary` violet (which remains the product brand color from feature 002). The token MUST meet WCAG AA contrast against both the light-mode background and the dark-mode background.
- **FR-008**: Negative monetary amounts MUST use a desaturated red derived from the existing `--destructive` token rather than a fully saturated red. The intent is to convey "negative" without conveying "error" — a credit-card balance of −$500 is normal, not alarming.
- **FR-009**: The brand mark MUST be exposed as a single contract surface (named `AbacusIcon` or equivalent) so that every place in the codebase that needs the brand mark renders the same component. The marketing header, the marketing footer, the sidebar brand area, the OG image, the favicon, and any empty-state illustration that uses the brand mark MUST all reference this single contract surface or its underlying source so that visual drift across surfaces is structurally prevented.

#### Money typography

- **FR-010**: The system MUST introduce a money rendering primitive (named `Money` or equivalent) in the UI layer, distinct from the `lib/money/` arithmetic layer established by feature 004. The primitive's role is rendering only — it MUST NOT perform any arithmetic on the amount it receives. The primitive's contract MUST accept at minimum a Decimal-valued amount and a currency code, plus rendering hints (size, alignment, whether to show the currency code prominently); the exact prop names and shape are plan-level.
- **FR-011**: The money rendering primitive MUST use tabular numerals so that digits in different rows align vertically in a column. This is the most visible single trust signal in the money UI.
- **FR-012**: The money rendering primitive MUST render the amount together with its currency in 100% of rendered states. The primitive MUST NOT accept a configuration that suppresses the currency entirely. The primitive MAY render the currency at a smaller size or muted treatment alongside the amount, but the currency MUST always be present. This is the rendering-layer enforcement of constitution Principle I (reaffirmed by feature 004 FR-011).
- **FR-013**: The money rendering primitive MUST apply sign-aware color according to the documented policy: positive amounts use the default `foreground` color; zero amounts use the `muted-foreground` color; negative amounts use the desaturated red derived from `--destructive` (FR-008). The new `money-positive` token (FR-007) MUST NOT be applied by the rendering primitive in this chore — it is reserved for explicit income / gain semantics introduced by future features (007 Dashboard, 008 Budgets, 015 Charts). The visible sign character (`+`, `−`) MUST remain present so that color is NOT the sole carrier of sign information.
- **FR-014**: The accounts list (feature 004's primary surface) MUST be migrated to render its balance column through the money rendering primitive. The balance column MUST be right-aligned. The other columns (Name, Type, Currency) remain left-aligned. The balance column MUST become the visually most prominent column on the page (achieved through typography weight, color contrast, or both).
- **FR-015**: The money rendering primitive MUST be the single contract surface for monetary display across all surfaces in the codebase. After this chore lands, any future feature that displays money (feature 006 Transactions, feature 007 Dashboard, feature 008 Budgets, feature 015 Charts, etc.) MUST render through this primitive. This rule is documented in the spec; the planner is free to add a lint or convention to enforce it.

#### Dashboard footer

- **FR-016**: The system MUST introduce a footer rendered at the bottom of every authenticated route (`/dashboard`, `/dashboard/accounts`, `/dashboard/transactions`, `/dashboard/budgets`, `/dashboard/settings`). The footer is named `ShellFooter` or equivalent and is a contract surface mounted once in the authenticated shell layout so that every authenticated route inherits it without per-route wiring.
- **FR-017**: The shell footer MUST be content-minimal: at least the abacus brand mark, the wordmark, and a short attribution line (e.g., a copyright statement and/or a version string). The footer MUST NOT replicate the marketing footer's full link farm; it is a brand reaffirmation, not a navigation surface.
- **FR-018**: The shell footer MUST be pinned to the bottom of the viewport on pages whose content is shorter than the viewport (sticky-bottom behavior). On pages whose content exceeds the viewport, the footer MUST sit at the natural end of the scrollable content. The footer MUST NOT cover or visually conflict with the mobile drawer when the drawer is open.
- **FR-019**: The theme toggle MUST remain reachable from every authenticated route on every viewport width supported by feature 002 (320px to 2560px). The exact location of the toggle (in the top header, in the new footer, or in both) is plan-level; the spec only locks the accessibility-from-everywhere bar. If the toggle is placed in the footer, mobile discoverability MUST be preserved.

#### Empty states

- **FR-020**: The system MUST upgrade the existing `EmptyState` contract surface (established by feature 002) to support an illustration slot in addition to its existing `icon` prop. The illustration slot accepts a richer brand-consistent visual (stroke-based, monochrome with one accent — consistent with the brand mark aesthetic). The existing `icon` prop MUST continue to work so that any code path passing only an icon does not break; the illustration slot takes precedence when both are provided.
- **FR-021**: The upgraded `EmptyState` contract surface MUST additionally support a decorative preview slot — a faded, non-interactive mock of what the loaded experience will look like once data is present. The preview MUST be announced as decorative to assistive technology (i.e., excluded from the accessibility tree) and MUST NOT be part of the keyboard tab order.
- **FR-022**: The `/dashboard` route MUST render a welcome panel using the upgraded `EmptyState` contract surface, with the abacus brand mark as the illustration. The panel's copy MUST name what is available today (Accounts) and what is coming (Transactions, Budgets — and other roadmap features as a general gesture, not a commitment). The panel MUST include a primary call-to-action that links to `/dashboard/accounts`. The existing disabled "Add your first account" affordance and the legacy "future feature" caption MUST be removed.
- **FR-023**: The `/dashboard/accounts` route's zero-accounts empty state MUST be migrated to the upgraded `EmptyState` contract surface with a brand-consistent illustration. The existing call-to-action behavior (opens the create side sheet) from feature 004 MUST be preserved without regression.
- **FR-024**: The `/dashboard/transactions` route MUST render a coming-soon empty state using the upgraded `EmptyState` contract surface, including a brand-consistent illustration, a route-specific headline (e.g., "Transactions are coming soon"), a one-line description of what the feature WILL DO (e.g., "Track every dollar in and out of your accounts, categorise them, and see where your money goes"), and a decorative faded preview of a two-row transaction-table layout below the description. The empty state MUST NOT name a roadmap feature number, MUST NOT link to an external roadmap, and MUST NOT include a release-cadence commitment. No primary call-to-action button is rendered (no destination would be honest).

## Assumptions

- Feature 002's app shell, feature 003's auth boundary, and feature 004's accounts feature are all shipped to `main` and in their current state. This chore builds on top of those features and does not regress any of their FRs or SCs.
- The brand color violet established by feature 002 (mapped to shadcn's `--primary` CSS variable) remains the product's primary brand color. The new `money-positive` green is a SECOND accent color used only for monetary display; it does not replace violet anywhere.
- The brand typeface choice is Inter (or an equivalent neutral, geometric, screen-optimized sans-serif). The exact typeface and the exact font-display strategy are plan-level decisions; the spec locks only the perceptual bar ("no obvious flash" and "graceful system-stack fallback").
- The brand mark is a single contract surface used everywhere the mark renders. The plan stage decides whether the mark is an SVG file, an inline React component, or both; the spec locks only the single-source-of-truth rule (FR-009).
- The OG image and the favicon are generated through the project framework's built-in conventions for those artifacts. No third-party OG-image service and no third-party favicon-generator is introduced by this chore.
- Inter (or equivalent) loads through the project framework's built-in font convention so that no third-party font-CDN runtime request is required. This satisfies FR-005's self-hosting requirement structurally rather than through a custom configuration.
- The existing `lib/money/` arithmetic layer from feature 004 is unchanged. The new money rendering primitive is a UI-layer wrapper that consumes the existing formatter from `lib/money/`; it does not duplicate or replace the formatting logic that already lives there.
- The existing `EmptyState` primitive from feature 002 is the single empty-state contract surface across the app. This chore extends it (illustration slot, preview slot) rather than introducing a parallel primitive.
- The accounts table's `Name`, `Type`, and `Currency` columns retain their existing left-alignment from feature 004. Only the `Balance` column changes (right-alignment, tabular numerals, sign-aware color, currency-adjacent rendering).
- Coming-soon empty states for `/dashboard/transactions` and `/dashboard/budgets` are valid through the lifetime of this chore. Feature 006 (Transactions) and feature 008 (Budgets), when they ship, will replace these empty states with their own loaded content; the empty-state primitive contract is preserved across that handoff.
- The dashboard welcome panel introduced by FR-022 is an interim. Feature 007 (Real dashboard with net-worth, this-month cash flow, recent transactions) will replace this panel with real widgets when it ships. The chore's welcome panel is correctly aligned with the product's current capability set (Accounts only); it does not pre-promise feature 007's widgets.
- The chore does NOT introduce a `/styleguide` design-system route. The current set of UI primitives is small enough that a documentation surface would be overhead, not leverage.
- The chore does NOT introduce additional dark/light theme variants beyond the existing two from feature 002. Users continue to pick light, dark, or system via the existing theme toggle.
- The chore does NOT introduce any audit-log model of theme preference. Theme remains browser-local (via `next-themes` from feature 002).

## Out of Scope

- **Real dashboard widgets (net worth, this-month cash flow, recent transactions, quick-add CTAs)** — these are feature 007 (Real dashboard) on the roadmap and are NOT introduced by this chore. This chore's `/dashboard` content is a branded welcome panel only.
- **Charts / data visualization / Recharts** — feature 015 (Charts) on the roadmap. No chart is introduced by this chore.
- **New shadcn primitives** — this chore uses only the shadcn primitives already on file. No new shadcn component is added.
- **New routes** — every existing route is preserved with the same URL; their content evolves visually, but the middleware behavior and the route set are unchanged.
- **Database schema changes** — no new Prisma migration. No new domain entity. No new column on any existing entity.
- **New runtime dependencies** — the chore uses only what the project framework provides natively (built-in font convention, built-in OG-image convention) and what is already on file (shadcn primitives, lucide icons). No third-party illustration library, animation library, icon pack, font CDN, OG-generator service, or favicon-generator service is added.
- **Visual-regression test infrastructure** — Playwright screenshot diffing, Chromatic, Percy, or any similar tool. Existing E2E tests cover behavior and MUST continue to pass; this chore does NOT add pixel-perfect visual coverage.
- **A `/styleguide` or design-system documentation route** — overhead exceeds the leverage given the current size of the UI primitive set.
- **Additional theme variants beyond light/dark/system** — the two themes from feature 002 are sufficient.
- **An audit log of which user prefers which theme** — theme is browser-local; this chore does not introduce any per-user theme persistence on the server.
- **A "What's new" or in-app changelog widget on the dashboard** — the marketing changelog stays on `/`; the dashboard welcome panel MAY link to it, but no dashboard-resident changelog widget is introduced.
- **Per-route detail pages for accounts, transactions, or budgets** — out of scope and unrelated to this chore.
- **Internationalization of any of the new copy** — all new strings are English-only; locale-switching infrastructure is deferred.

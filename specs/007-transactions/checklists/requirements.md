- Clarification session 2026-05-17 resolved all three originally-flagged questions. Load-bearing weight from highest to lowest:
  - **Q1 — amount sign convention → SIGNED AMOUNTS.** This is the most load-bearing decision in the spec. It defines the schema column type (signed Decimal), the boundary validation (sign-must-match-type), the balance-computation formula (a single SUM, not a CASE-WHEN), and the downstream contracts for features 007 (dashboard net-worth widget), 008 (budget rollups), 015 (charts), and 016 (reports). Getting it wrong would force a destructive migration with a full-table backfill; getting it right means the formula `startingBalance + Σ(amount)` is correct for every aggregation surface in the app. Rationale documented in the Clarifications session and FR-008.
  - **Q3 — hard delete vs soft delete → SOFT DELETE (ARCHIVE).** Second-most load-bearing. It defines the deletion semantics for the third domain entity (after Accounts and Categories), keeps the three entities consistent (same `archivedAt` pattern, same "Show archived" toggle, same archive-cascade-on-transfer rule analogous to no-cascade-on-category), and makes future audit-log integration (feature 021) and optional CSV-export-with-archived (feature 014) straightforward. Hard delete remains explicitly out of scope; a future polish feature MAY add it. Rationale documented in Clarifications and FR-017 / FR-018.
  - **Q2 — date semantics → CALENDAR-DAY-ONLY (UTC midnight).** Least load-bearing of the three but still non-trivial. Locks deterministic same-day sort, intuitive date-range filters ("this month" is a clean inclusive range, not a timezone-shifted timestamp range), and trivially correct per-day aggregation for features 015 and 016. The user's audit timestamps (`createdAt`, `updatedAt`) remain full-precision. Rationale documented in Clarifications and FR-004.

# Feature Specification: Transactions + Transfers

**Feature Branch**: `007-transactions`

**Created**: 2026-05-17

**Status**: Draft

**Input**: User description: "for Transactions" (kicked off via `/speckit-specify` after Categories merged)

> **Numbering note**: This is the roadmap-numbered feature **006 — Transactions + Transfers** (Tier 1). The spec directory is `007-transactions` because the branded-UI polish chore consumed spec-dir slot `005-branded-ui-polish` in May 2026. The roadmap entry and this spec describe the same feature; the directory number tracks order-of-spec-authored, not the roadmap slot.

## Why

Transactions are the actual money flow that the rest of Abacus has been building toward. Until this feature ships, the app knows about accounts (feature 004) and categories (feature 005) but has nowhere for the user to record that they paid rent, received a salary, transferred from checking to savings, or bought groceries. Every later money-touching feature stacks directly on top of this one: roadmap feature 007 (real dashboard widgets — net worth, this-month cash flow, recent transactions) reads from it, feature 008 (Budgets) compares against it, feature 009 (Search & filter) filters it, feature 010 (CSV import) writes into it, feature 015 (Charts) aggregates it, feature 016 (Reports) reports against it, and feature 020 (multi-currency FX) extends transfers across currencies. This is the keystone of the Tier 1 MVP.

This is also the first feature in which constitution Principle I (money math) leaves the spec stage and enters the runtime. Feature 004 laid the `lib/money/` boundary and stored a single Decimal value (`startingBalance`) per account; feature 006 turns that boundary into a working arithmetic surface — multiple monetary writes per account per day, transfer pairs that MUST be atomic, balance computations that MUST equal `startingBalance + Σ(transactions)` byte-for-byte. The money-reviewer subagent will audit this PR; nothing in this spec may make their job harder.

## Clarifications

### Session 2026-05-17

- Q: Amount sign convention — always-positive with the `type` carrying the sign, or signed amounts where `type` is informational? → A: **Signed amounts.** The `amount` Decimal is positive for INCOME, negative for EXPENSE, and one signed pair for TRANSFER (the source leg negative, the destination leg positive). The `type` enum is informational and is validated at the boundary to match the sign (a payload that says `type=INCOME` with `amount=-50.00` is rejected; a payload that says `type=EXPENSE` with `amount=+50.00` is rejected). Rationale: this makes the per-account balance computation a single SUM (`startingBalance + Σ(amount where archivedAt is null)`) instead of a CASE-WHEN over type, which is what feature 007's net-worth widget, feature 008's budget rollups, feature 015's charts, and feature 016's reports all want. It also matches double-entry accounting and the Firefly III model. The boundary validation cost (sign-must-match-type) is trivial and lives in one Zod schema. This decision is load-bearing for the schema and four downstream features; locking it now prevents a destructive migration later.
- Q: Date semantics — calendar-day-only (DateTime normalized to 00:00:00 UTC of that day) or full timestamp with the user's submitted time preserved? → A: **Calendar-day-only.** The `date` column is a `DateTime` whose time component is normalized to 00:00:00 UTC of the calendar day the user records. Personal finance is calendar-day-granular; users do not care WHEN on 2026-05-17 they bought coffee, only THAT it was 2026-05-17. Normalizing at the boundary keeps sorting deterministic (no two-transactions-on-the-same-day rendering inconsistency from sub-second drift), keeps date-range filters intuitive (a "this month" filter is a calendar-day range, not a timezone-shifted timestamp range), and keeps future per-day aggregation (feature 015's charts, feature 016's reports) trivially correct. The user's audit timestamps (`createdAt`, `updatedAt`) remain full-precision; only the user-visible `date` is calendar-day-normalized.
- Q: Hard delete vs soft delete (archive) — should transactions be hard-deleted like a typo or soft-deleted like accounts and categories? → A: **Soft delete (archive).** A transaction or transfer is archived, not removed, via the same `archivedAt` pattern feature 004 (Accounts) and feature 005 (Categories) use. Default lists exclude archived rows; a "Show archived" toggle reveals them. Rationale: consistency with the two prior domain entities is high-value; storage is cheap; soft delete is undo-friendly on a money-touching surface where a destructive action with no recovery would feel hostile; future features (roadmap feature 021 audit log, feature 014 CSV export with optional inclusion of archived rows) can lean on the archived-but-retained data. Hard delete is explicitly OUT OF SCOPE for v1; a future polish feature MAY add it with safeguards (e.g., only transactions younger than N days). Archive on a transfer leg MUST archive BOTH legs atomically (per the two-leg invariant); the same is true of unarchive.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Record an expense transaction (Priority: P1)

As a user who has just bought groceries and wants to track the $87.43 I spent at Whole Foods, I open the transactions page, activate the "Add transaction" CTA, fill in the account I paid from, the category ("Groceries"), the date (today by default), the amount, optionally a payee ("Whole Foods") and notes, submit, and immediately see the new row in the list with its negative amount shown in the account's currency, AND see the source account's balance decrease by exactly $87.43.

**Why this priority**: This is the single most common interaction in a personal-finance app — the day-to-day "I spent money" flow. Without it, the app is a static accounts list with no money flow. Every user touches this path multiple times a week; every later feature (Budgets, Reports, Charts) is meaningless without a stream of expense transactions to operate on. P1 because the create-expense flow alone is the smallest viable money-flow demo of Abacus.

**Independent Test**: From a user with at least one account in USD and at least one EXPENSE category seeded, navigate to `/dashboard/transactions`. Activate the primary "Add transaction" CTA. A side sheet opens. Fill `account = "Chase Checking"`, `category = "Groceries"`, `date = today`, `amount = 87.43`, `type = EXPENSE`, `payee = "Whole Foods"`, leave notes blank, submit. The sheet closes; the new row appears in the list, dated today, with the negative amount and the USD currency rendered together. Navigate to `/dashboard/accounts`; the Chase Checking row's balance is now `startingBalance - 87.43`.

**Acceptance Scenarios**:

1. **Given** an authenticated user with at least one account and at least one EXPENSE category, **When** they open `/dashboard/transactions` and activate "Add transaction", **Then** a side sheet opens in "create" mode with `account` empty, `category` empty, `date` defaulted to today (in the user's timezone, normalized at submit to UTC calendar-day), `amount` empty, `type` defaulted to `EXPENSE`, `payee` empty, `notes` empty.
2. **Given** the create sheet is open, **When** the user submits a valid expense (account, category, date, amount, payee, notes — with `type=EXPENSE` and `amount > 0` entered by the user; the boundary stores `amount` as a negative Decimal per the signed-amount convention), **Then** the transaction is persisted, the sheet closes, and the new row appears in the list at the top (sorted by date descending) with its negative amount and the account's currency rendered together via the `<Money>` primitive.
3. **Given** the user just created an expense for $87.43 on Chase Checking, **When** they navigate to `/dashboard/accounts`, **Then** the Chase Checking row's displayed balance is `startingBalance - 87.43` rendered in the account's currency.
4. **Given** the user submits an expense, **When** they reload `/dashboard/transactions`, **Then** the same transaction is still listed (the data persisted, scoped to this user).
5. **Given** a SECOND authenticated user has signed up and is on `/dashboard/transactions`, **When** the first user created the expense in scenario 2, **Then** the second user's list is empty — they see only the empty state and see none of the first user's transactions.

### User Story 2 - Record an income transaction (Priority: P1)

As a user who has just been paid my $5,000 salary and wants to record it as income against my "Salary" category in Chase Checking, I open the transactions page, activate "Add transaction", fill in the account, the category, the date, the amount, optionally the payee ("Acme Corp") and notes, submit, and immediately see the new row with its positive amount rendered in the account's currency AND see the source account's balance increase by exactly $5,000.

**Why this priority**: Income is the symmetric counterpart to expense — every user receives money as well as spends it, and the app must represent both directions natively. Without it, balances drift monotonically downward and the per-category aggregation (which feature 008 budgets, feature 015 charts, and feature 016 reports will all consume) is incomplete on the income side. P1 because the create-income flow is co-equal with create-expense in user value — they are the two halves of the same primitive.

**Independent Test**: From a user with at least one account and the default INCOME categories seeded (e.g., "Salary"), navigate to `/dashboard/transactions`. Activate "Add transaction". Set `type = INCOME` in the form. Fill `account = "Chase Checking"`, `category = "Salary"`, `date = today`, `amount = 5000.00`, `payee = "Acme Corp"`, leave notes blank, submit. The sheet closes; the new row appears in the list with its positive amount and the USD currency rendered together. The Chase Checking account's balance increases by $5,000.

**Acceptance Scenarios**:

1. **Given** the create sheet is open and the user changes `type` from EXPENSE to INCOME, **When** the category picker re-renders (or the user opens it), **Then** the category picker lists the user's INCOME categories (and only INCOME categories) at the top, with EXPENSE categories visually deprioritized or hidden — the picker reflects the kind constraint so the user can't accidentally categorize an income transaction with an expense category (FR-009).
2. **Given** the create sheet is open with `type=INCOME`, **When** the user submits a valid income (account, INCOME category, date, amount, payee, notes — with `amount > 0`; the boundary stores `amount` as a positive Decimal per the signed-amount convention), **Then** the transaction is persisted, the sheet closes, the new row appears in the list with its positive amount and the account's currency rendered together.
3. **Given** the user just created income for $5,000 on Chase Checking, **When** they navigate to `/dashboard/accounts`, **Then** the Chase Checking row's displayed balance is `startingBalance + 5000` rendered in the account's currency.
4. **Given** the user submits `type=INCOME` paired with `amount` entered as a negative number (e.g., user typed a minus sign by mistake), **Then** the form rejects the submission at the boundary with an actionable error explaining that income must be entered as a positive amount (FR-008). The persisted state is unchanged.

### User Story 3 - Transfer money between two accounts (Priority: P1)

As a user who has just moved $500 from my Chase Checking to my Savings, I open the transactions page, activate the "Add transfer" CTA (a SEPARATE primary action from "Add transaction"), fill in the source account, the destination account, the date, the amount, optionally notes, submit, and immediately see the transfer reflected as a single conceptual row in the list (or two paired rows; rendering is plan-level) AND see Chase Checking's balance decrease by $500 AND Savings' balance increase by $500 — with the system guaranteeing that BOTH halves of this transfer were written in a single atomic database transaction (if either half had failed, neither would have persisted).

**Why this priority**: Transfers are the single feature most personal-finance apps get wrong, and the constitution calls them out explicitly: a transfer MUST be atomic — both ledger entries (debit on source, credit on destination) MUST be written in one DB transaction, or neither persists. P1 because skipping atomicity is the most common money-correctness bug in the entire app category (a "transfer" that succeeds on one side and fails on the other silently corrupts both balances), and because once feature 006 ships, every later feature treats the transfer-pair invariant as a given. Locking this in v1 is non-negotiable.

**Independent Test**: From a user with at least two accounts of the same currency (e.g., Chase Checking USD and Savings USD), navigate to `/dashboard/transactions`. Activate the "Add transfer" CTA. The sheet that opens is DIFFERENT from the "Add transaction" sheet — it has `From account`, `To account`, `Date`, `Amount`, optional `Notes`, and no `Category` field, no `Type` field, no `Payee` field. Fill `From = Chase Checking`, `To = Savings`, `Date = today`, `Amount = 500.00`, leave notes blank, submit. The sheet closes; the transfer is reflected in the list. Navigate to `/dashboard/accounts`; Chase Checking is `startingBalance - 500`, Savings is `startingBalance + 500`. Inspect persisted state (e.g., via a list filtered to type=TRANSFER): there are exactly two transaction rows that share the same `transferGroupId`, the same `date`, the same `currency`, the same `userId`, AND whose `amount` values are exact inverses (`-500.00` on Chase Checking, `+500.00` on Savings).

**Acceptance Scenarios**:

1. **Given** the user is on `/dashboard/transactions`, **When** they look at the page header, **Then** there are TWO primary CTAs: "Add transaction" (for INCOME / EXPENSE) and "Add transfer" (for TRANSFER). The two CTAs are visually distinguishable; activating each opens a DIFFERENT side sheet with a different form shape (FR-021).
2. **Given** the transfer sheet is open, **When** the form renders, **Then** the fields are `From account` (picker), `To account` (picker), `Date` (defaulted to today), `Amount` (entered as positive Decimal), and optional `Notes`. There is no `Category` field (transfers are inherently uncategorized — they reclassify existing money, they neither earn nor spend), no `Type` field (it is implicitly `TRANSFER`), and no `Payee` field.
3. **Given** the transfer sheet is open with `From = Chase Checking (USD)` and `To = Savings (USD)`, **When** the user submits a valid transfer (amount > 0, From ≠ To, both accounts owned by this user, both accounts non-archived, both currencies equal), **Then** the system creates exactly TWO `Transaction` rows in a single atomic database transaction: one with `accountId = ChaseCheckingId, type = TRANSFER, amount = -500.00`; one with `accountId = SavingsId, type = TRANSFER, amount = +500.00`. Both rows share the same generated `transferGroupId`. Both rows share the same `date`. Both rows share the same `currency`. Both rows share the same `userId`. If either insert had failed for any reason (DB error, validation failure on the second leg), neither row persists (FR-013).
4. **Given** the user has just transferred $500 from Chase Checking to Savings, **When** they navigate to `/dashboard/accounts`, **Then** Chase Checking's displayed balance is `startingBalance - 500` and Savings' displayed balance is `startingBalance + 500`, both rendered in their native currency via the `<Money>` primitive.
5. **Given** the transfer sheet is open with `From = Chase Checking` and `To = Chase Checking` (same account on both sides), **When** the user submits, **Then** the form rejects the submission at the boundary with an actionable error explaining that source and destination must be different accounts (FR-014). The persisted state is unchanged.
6. **Given** the transfer sheet is open with `From = Chase Checking (USD)` and `To = EuroSavings (EUR)` (different currencies), **When** the user submits, **Then** the form rejects the submission at the boundary with an actionable error explaining that cross-currency transfers are not supported in this version (FX conversion lands in roadmap feature 020), and the persisted state is unchanged (FR-015). Zero rows are created.
7. **Given** the transfer sheet is open with `From = Chase Checking` and the user has archived the Savings account, **When** the user opens the `To account` picker, **Then** archived accounts MUST NOT appear in the picker. A new transfer cannot route to an archived account (FR-022). The same rule applies to the `From account` picker.

### User Story 4 - Edit or archive a transaction or transfer (Priority: P1)

As a user who mistyped the amount on yesterday's grocery expense, or who realized I categorized rent under "Other Expenses" when I meant "Housing", or who needs to back out a transfer I recorded twice, I can click any row in the transactions list, see the existing values pre-populated in a side sheet, change the fields I want to change, save — and the row updates immediately. For an archive, I activate "Archive", confirm, and the row disappears from the default list. For a transfer, editing or archiving operates atomically on BOTH legs of the pair, NEVER on one leg in isolation.

**Why this priority**: Personal finances accumulate mistakes — typos, mis-categorizations, duplicate entries, dates entered in the wrong month. Without edit and archive, the create-only flow from US1–US3 becomes a one-way trap and users abandon the app the first time they catch themselves making a typo. P1 because the create flow alone is not a viable MVP, AND because the transfer-pair atomicity invariant applies symmetrically on update and archive — getting it right on create and wrong on update would corrupt balances silently, which is the failure mode the constitution exists to prevent.

**Independent Test**: From a user with at least one INCOME or EXPENSE transaction and at least one transfer, navigate to `/dashboard/transactions`. Click a single-leg transaction (INCOME or EXPENSE). A side sheet opens in "edit" mode pre-populated with the current account, category, date, amount, type, payee, notes. Change the amount, save: the row updates AND the linked account's balance recomputes. Reopen the same row, activate "Archive", confirm: the row leaves the default list and the account balance recomputes (the archived transaction is excluded from the SUM). Toggle "Show archived": the row reappears with an "Archived" badge. Click a transfer row in the list. The sheet that opens shows the transfer as a UNIT — `From account`, `To account`, `Date`, `Amount`, `Notes` — NOT a single leg with one accountId; the user cannot edit just one leg in isolation. Change the amount on the transfer, save: BOTH legs are updated atomically (both legs' `amount` Decimals are recomputed; both legs share the new value's magnitude in the correct signs; both balances on both accounts recompute correctly). Reopen the transfer, activate "Archive", confirm: BOTH legs are archived atomically (both rows' `archivedAt` is set in one DB transaction). Toggle "Show archived": both legs of the transfer reappear together, both with "Archived" badges.

**Acceptance Scenarios**:

1. **Given** the user has at least one single-leg transaction (INCOME or EXPENSE), **When** they click that row in the list, **Then** a side sheet opens in "edit" mode pre-populated with the current `account`, `category` (nullable), `date`, `amount` (shown as the user-entered positive magnitude with the sign implied by the type), `type`, `payee`, `notes`.
2. **Given** the single-leg edit sheet is open, **When** the user changes any editable field (account, category, date, amount, type, payee, notes) to a valid value and submits, **Then** the transaction's record updates, the sheet closes, the list row reflects the change, and the affected account's displayed balance recomputes per the formula (FR-016).
3. **Given** the single-leg edit sheet is open, **When** the user changes the `account` to a different account of the SAME currency, **Then** the transaction's `accountId` updates, the OLD account's balance recomputes (subtracting this transaction's amount) AND the NEW account's balance recomputes (adding this transaction's amount), in 100% of attempts.
4. **Given** the single-leg edit sheet is open, **When** the user attempts to change the `account` to a different account of a DIFFERENT currency, **Then** the submission is rejected at the boundary with an actionable error (a transaction's currency MUST match its account's currency — FR-007). The persisted state is unchanged.
5. **Given** the single-leg edit sheet is open, **When** the user activates "Archive" and confirms, **Then** the transaction's `archivedAt` is set, the sheet closes, the row disappears from the default list view, and the account's balance recomputes excluding this archived transaction (FR-017).
6. **Given** the user has at least one archived transaction, **When** they enable the "Show archived" toggle, **Then** the archived row appears in the list with an "Archived" badge and a muted treatment. Active rows remain visible above (or in the usual date-descending position).
7. **Given** the user clicks a TRANSFER row in the list, **When** the sheet opens, **Then** it opens in "transfer edit" mode — a DIFFERENT form from the single-leg edit form — pre-populated with `From account`, `To account`, `Date`, `Amount` (the positive magnitude), `Notes`. The user CANNOT see or edit a single leg in isolation (FR-024); the two legs are always presented as one transfer.
8. **Given** the transfer-edit sheet is open, **When** the user changes the amount (or date, or notes, or swaps the From/To accounts) to a valid value and submits, **Then** the system updates BOTH legs in a single atomic database transaction: both legs share the new `date`, both legs share the new `amount` magnitude (one stored negative on the source leg, one stored positive on the destination leg per the signed-amount convention), both legs share the new `notes`, both legs share the same `transferGroupId` (unchanged). If either leg's update fails, neither leg's update persists (FR-013).
9. **Given** the transfer-edit sheet is open, **When** the user activates "Archive" and confirms, **Then** BOTH legs of the transfer are archived in a single atomic database transaction (both `archivedAt` set to the same timestamp). Toggling "Show archived" reveals BOTH legs together; archiving one without the other is impossible by construction (FR-018).
10. **Given** a user owns transactions A and B, and a SECOND user owns transaction C, **When** the first user submits an edit payload that references C by id (single-leg) or `transferGroupId` (transfer), **Then** the operation MUST fail with a not-found-style error envelope (FR-013); the second user's transaction MUST NOT be reachable through any product surface to the first user.

### User Story 5 - Filter and scan the transactions list (Priority: P2)

As a user with hundreds of transactions accumulated over several months, I need to find the ones I care about right now — the expenses in May 2026, or every transaction on Chase Checking, or every transaction categorized "Groceries", or every transfer — without scrolling through everything I have ever recorded. I open the transactions page, pick a date range, pick an account filter (or "All accounts"), pick a category filter (or "All categories" / "Uncategorized"), pick a type filter (or "All types"), optionally toggle "Show archived", and the list re-renders to show only the matching rows, sorted by date descending.

**Why this priority**: A flat unfiltered list becomes unusable around 50 transactions. Without filters, the user cannot answer "how much did I spend on Groceries this month?" or "did I record that transfer last week?" — both of which are core personal-finance questions. P2 because US1–US4 (create + edit + archive) are functionally usable on a small dataset without filters, but the moment a user has even a single month of data the unfiltered list stops being scannable. The filter set defined here is also the contract feature 009 (Search & filter) will extend with free-text search; locking the basic filter shape in v1 means feature 009 is additive, not a rework.

**Independent Test**: From a user with at least 10 transactions spread across multiple accounts, categories, types, and dates, navigate to `/dashboard/transactions`. The default view shows the last 30 days (a sensible default; the user can extend the range). Activate the date-range picker and set a specific range (e.g., "2026-05-01 to 2026-05-31"); the list re-renders to only the transactions in that range. Set the account filter to a single specific account; the list further narrows. Set the category filter to "Groceries"; the list narrows again. Set the type filter to "EXPENSE"; the list narrows again. Toggle "Show archived" on; archived rows in the matching set appear with the "Archived" badge. Reset every filter to its default; the list returns to the default view (last 30 days, all accounts, all categories, all types, show archived off).

**Acceptance Scenarios**:

1. **Given** the user navigates to `/dashboard/transactions`, **When** the page loads, **Then** the default filter state is: date range = last 30 days (inclusive of today, the previous 30 calendar days), account = all, category = all, type = all, show archived = off. The list renders the matching rows sorted by date descending (most recent first), with a stable secondary sort by `createdAt` descending (FR-020).
2. **Given** the user adjusts the date-range picker to an explicit start and end, **When** the form re-renders, **Then** the list shows only transactions whose `date` falls within the inclusive range, sorted by date descending.
3. **Given** the user selects a specific account in the account filter, **When** the form re-renders, **Then** the list shows only transactions whose `accountId` matches (including both legs of a transfer whose `accountId` matches that account — the OTHER leg of the same transfer, on a different account, is excluded from the filtered view; this is consistent with the formula "this account's transactions = all rows with this `accountId`").
4. **Given** the user selects a specific category in the category filter, **When** the form re-renders, **Then** the list shows only transactions whose `categoryId` matches. Selecting "(uncategorized)" filters to transactions where `categoryId` is null. Transfers (which have `categoryId = null` by FR-006) are included only when the "(uncategorized)" filter is active or when type=TRANSFER is selected with category=all.
5. **Given** the user selects a specific type in the type filter, **When** the form re-renders, **Then** the list shows only transactions whose `type` matches the selected value (INCOME, EXPENSE, or TRANSFER).
6. **Given** the user enables "Show archived", **When** the form re-renders, **Then** archived transactions in the matching filter set appear in the list alongside active ones, with a clear "Archived" badge and a muted treatment.
7. **Given** the user has applied a combination of filters and the matching set has MORE transactions than fit in the initial render window, **When** the user scrolls to the bottom or activates a "Load older" affordance, **Then** older matching transactions become visible (FR-026). The exact mechanism — paginated, infinite-scroll, or "load older" button — is plan-level, but the user can reach every matching row in the current filter set.
8. **Given** the user resets every filter to its default (date range → last 30 days, account → all, category → all, type → all, show archived → off), **When** the form re-renders, **Then** the list returns to the default view exactly as it was on first load (FR-019).

### User Story 6 - Validation surfaces actionable errors on every money-touching field (Priority: P2)

As a user filling out the create or edit form for a transaction or a transfer, if I submit invalid input — a blank amount, a non-numeric amount, an amount with more decimal places than the currency supports, a negative amount on the wrong sign-convention side, a missing account, a missing date, a category whose kind does not match the type, a transfer where source and destination are the same account, a transfer across currencies, a payee or notes string that exceeds the maximum length, a transaction whose stated currency does not match the account's currency — the form rejects the submission at the boundary, shows me a clear field-scoped error explaining what is wrong, keeps my other entered values intact so I can fix the one offending field, and persists ZERO changes.

**Why this priority**: This is a money-touching surface. Validation gaps here corrupt balances — an EXPENSE with `amount > 0` would silently increase a balance instead of decreasing it; a transfer that survives only its source-leg validation would withdraw from one account without depositing into the other. Constitution Principle III mandates that validation happens at the boundary before any business logic; constitution Principle IV mandates that the money paths are tested. P2 because US1–US5 functionally pass with rudimentary "field is required" messages, but the load-bearing rules (sign-must-match-type, currency-must-match-account, transfer-same-currency, transfer-distinct-accounts, decimal-places-match-currency) are the gates between a clean ledger and silent corruption.

**Independent Test**: Submitting the create or edit form (single-leg or transfer) with each of the following invalid inputs surfaces a clear, field-scoped error and does NOT persist any change: blank amount, non-numeric amount, amount with more decimal places than the chosen account's currency supports (e.g., 3+ decimals for USD), `type=EXPENSE` with positive `amount`, `type=INCOME` with negative `amount`, missing account, missing date, date in an unsupported format, category whose kind does not match the type (e.g., INCOME transaction categorized with an EXPENSE category), payee longer than 120 characters, notes longer than 500 characters, transfer with source = destination, transfer with mismatched currencies, transaction whose payload's `currency` differs from the chosen account's `currency`, edit payload whose tampered body attempts to change `userId` or `transferGroupId`.

**Acceptance Scenarios**:

1. **Given** the create form is open, **When** the user submits with a blank or non-numeric `amount`, **Then** the form shows an actionable amount-field error and does not persist (FR-008).
2. **Given** the create form is open with `account = Chase Checking (USD)`, **When** the user submits with `amount = 87.4321` (4 decimal places, more than USD supports), **Then** the form shows an actionable amount-field error stating that the chosen currency supports at most 2 decimal places, and does not persist (FR-009).
3. **Given** the create form is open with `type = EXPENSE`, **When** the user submits with `amount` as a positive number in the input but the boundary stores it positive (forgetting the sign convention), **Then** the boundary either (a) auto-flips the sign at the Zod transformer per the convention so the stored value is negative, OR (b) rejects the payload with an actionable error. Whichever the plan picks, the post-persist invariant `(type=EXPENSE → amount < 0) AND (type=INCOME → amount > 0)` MUST hold for 100% of persisted rows (FR-008).
4. **Given** the create form is open, **When** the user submits with `account = empty` (no account selected), **Then** the form shows an actionable account-field error and does not persist (FR-005).
5. **Given** the create form is open with `account = Chase Checking (USD)` and `type = INCOME`, **When** the user picks a `category` whose `kind = EXPENSE` (e.g., "Groceries"), **Then** the form rejects the submission at the boundary with an actionable category-field error stating that an INCOME transaction requires an INCOME category, and does not persist (FR-011). The picker MAY also prevent the mismatched selection client-side (US2 scenario 1), but the boundary check is the authoritative gate.
6. **Given** the create form is open, **When** the user submits with `payee` longer than 120 characters, **Then** the form shows an actionable payee-field error and does not persist (FR-010).
7. **Given** the create form is open, **When** the user submits with `notes` longer than 500 characters, **Then** the form shows an actionable notes-field error and does not persist (FR-010).
8. **Given** the transfer create form is open, **When** the user submits with `From = Chase Checking` and `To = Chase Checking` (same account), **Then** the form shows an actionable error scoped to the `To account` field and does not persist (FR-014). Zero rows are created.
9. **Given** the transfer create form is open, **When** the user submits with `From = Chase Checking (USD)` and `To = EuroSavings (EUR)` (mismatched currencies), **Then** the form shows an actionable error stating that cross-currency transfers are not supported in this version and does not persist (FR-015). Zero rows are created.
10. **Given** a transaction's `account` is Chase Checking (USD) and the user submits an edit payload whose body's `currency` field is `EUR` (tampered), **Then** the boundary rejects the payload with an actionable error (FR-007); the currency of any transaction MUST equal the currency of its account, and the boundary enforces this on every write.
11. **Given** the form has just been rejected on the server, **When** the form re-renders, **Then** the entered values are preserved for the still-valid fields, the offending field is focused, and the user can correct it and re-submit (FR-027).
12. **Given** the user is editing an existing transaction or transfer, **When** any of the validation rules above fail on submit, **Then** the existing persisted row(s) are NOT modified (atomic save: all-or-nothing) (FR-013).

### Edge Cases

- **A user attempts to view, edit, archive, or unarchive a transaction that belongs to another user** (e.g., by guessing or tampering with an `id` or `transferGroupId` in a request) — the operation MUST fail with a not-found-style error envelope (FR-013). The system MUST NOT distinguish "this transaction belongs to another user" from "this transaction does not exist" in any user-visible response, log line, or response timing.
- **A user attempts to "delete" a transaction** — there is no hard-delete affordance in this feature. The only destructive-feeling action is "Archive", which is reversible (FR-017). This protects future audit-log integration (roadmap feature 021) and is consistent with feature 004 (Accounts) and feature 005 (Categories).
- **A user creates an EXPENSE with `amount = 0.00`** — rejected at the boundary with an actionable error: a transaction's `|amount|` MUST be strictly greater than zero (FR-008). A zero-amount transaction has no economic meaning and would not affect any balance; allowing it would pollute the list with empty rows.
- **A user creates a transaction whose `categoryId` is null (uncategorized)** — allowed for INCOME and EXPENSE (FR-006). Some users prefer to record the money flow first and categorize later; the list and filters surface "(uncategorized)" as a first-class filter option (US5 scenario 4). Transfers ALWAYS have `categoryId = null` by construction (FR-006); they are uncategorized by nature.
- **A user creates a transaction whose `categoryId` references a category they have since archived** — allowed; the transaction retains its `categoryId`. Archived categories are hidden from the picker on new transactions (FR-006a) but remain valid foreign-key targets on existing rows. The list renders such transactions with the archived category's name and a muted treatment on the category cell.
- **A user archives a category that is referenced by transactions** — feature 005's category archive is unaffected (categories archive as a soft state; existing references remain valid). The category cell on those transactions continues to render the archived category's name with a muted treatment. There is no cascade-archive on the transactions referencing the archived category.
- **A user attempts to delete an Account that is referenced by transactions** — the Account model uses archive-only (feature 004 FR-008); there is no hard-delete on accounts. The Account → Transaction foreign key uses RESTRICT semantics in the plan (this feature locks the rule; the migration is plan-level); but because there is no hard-delete UI on accounts, the user never reaches that path. Archiving an account does NOT cascade-archive its transactions (the historical record is preserved); the archived account is just hidden from default lists.
- **A user changes the `account` of an existing transaction from Chase Checking to Savings** — allowed for single-leg transactions only, and only if both accounts share the same currency (FR-007). The OLD account's balance recomputes (subtracting this transaction's amount); the NEW account's balance recomputes (adding it). For a transfer LEG, this kind of reassign is impossible by construction — the user edits the transfer as a unit, not a single leg (FR-024).
- **A user submits a transfer-edit payload that swaps `From` and `To`** (e.g., "I accidentally said Checking → Savings when I meant Savings → Checking") — allowed; the system rewrites BOTH legs atomically so the source leg's `accountId` and `amount` sign correspond to the new `From` and the destination leg's correspond to the new `To`. Both balances recompute correctly.
- **A user submits a transfer-edit payload that changes the amount from $500 to $200** — allowed; both legs' `amount` magnitudes update atomically (source leg now `-200.00`, destination leg now `+200.00`). Both balances recompute correctly.
- **A user submits a transfer-edit payload that attempts to change ONLY the source leg's account, leaving the destination leg untouched (via a tampered payload)** — impossible through the product surface (the transfer edit form operates on the pair as a unit, FR-024). A tampered API-level payload that tries to mutate only one leg by `id` is rejected at the boundary: any update to a row whose `transferGroupId` is non-null MUST go through the transfer-update path that mutates both legs together (FR-013).
- **A user creates a transaction dated in the future** — allowed in v1. The `date` field is the calendar day the user records; recording "next Tuesday's rent" today is a legitimate user workflow (forecasting). The list renders future-dated transactions normally; they appear in the user's default 30-day window if the window includes them, and they contribute to the account's displayed balance like any other transaction. (A future feature MAY introduce a "pending vs cleared" state; v1 does not.)
- **A user creates a transaction dated decades in the past** — allowed in v1. No upper or lower bound on the `date` field beyond standard `DateTime` storage. The user may be retroactively backfilling years of history; the app must support it.
- **A user submits an `amount` with a thousands separator or locale-specific decimal symbol** (e.g., "1.250,00" or "1,250.00") — the form normalizes at the boundary to the canonical Decimal form (decimal point, no thousands separator) before validation. Invalid numeric formats that cannot be normalized are rejected with an actionable error.
- **A user submits a transaction whose currency convention is zero-decimal (e.g., JPY)** — the amount-field decimal validation is currency-aware (FR-009), inherited from feature 004's currency-aware rule. JPY transactions reject any fractional input; BHD/KWD/JOD transactions permit three decimal places.
- **A user has no accounts when they navigate to `/dashboard/transactions`** — the page renders an empty state with a clear pointer to `/dashboard/accounts` ("Add an account first"); the create-transaction CTA is disabled (a transaction requires an account). This is consistent with feature 004 FR-010 (accounts is the on-ramp) and avoids the awkward state of a form that cannot submit (FR-029).
- **A user has accounts but no non-archived categories** — they CAN still create transactions (categories are optional per FR-006). The category picker shows an empty state or a pointer to `/dashboard/categories`. The transaction persists with `categoryId = null`.
- **A user with no session attempts to reach `/dashboard/transactions`** — covered by feature 003's auth boundary; the request is redirected to `/login` with the path preserved. This feature inherits that behavior and does not re-implement it.
- **A user with a session attempts to operate on a transaction by sending a tampered request body that includes a `userId` field** — the request body's `userId` (if present) is ignored. The session is the canonical source of identity (FR-003). Same for `transferGroupId` on create — the server generates it; a client-supplied `transferGroupId` is ignored (FR-018).
- **A user creates many transactions** — the list remains usable up to several thousand transactions via filtering + paginated rendering (FR-026). There is no enforced hard limit in v1.

## Requirements *(mandatory)*

### Functional Requirements

#### Data model

- **FR-001**: The system MUST introduce a `Transaction` domain model. At minimum the model MUST include a unique identifier, the owning user's identifier (foreign key to `User` with cascade-on-delete semantics), an `accountId` (foreign key to `Account` with restrict-on-delete semantics, because an account with transactions cannot be hard-deleted — feature 004 FR-008 already prevents account hard-delete from the UI, but the FK enforces it at the data layer), a nullable `categoryId` (foreign key to `Category` with restrict-on-delete semantics so a category referenced by a transaction cannot be hard-deleted — feature 005 FR-010 already prevents category hard-delete from the UI), a calendar-day `date`, a Decimal `amount`, a three-letter ISO 4217 `currency` code, a closed-enum `type` (`INCOME | EXPENSE | TRANSFER`), an optional `payee` string, an optional `notes` string, a nullable `transferGroupId` string (non-null for the two legs of a transfer; null for INCOME and EXPENSE rows), an optional `archivedAt` timestamp, and creation/updated-at timestamps. The model MUST land via a generated Prisma migration (no `db push` against committed code, per the constitution).
- **FR-002**: Every `Transaction` row MUST carry a `userId` foreign key referencing `User.id` with cascade-on-delete semantics, indexed by `userId`, so that removing a user removes their transactions in a single referential step. This is the third domain entity to exercise feature 003's data-scoping convention (after `Account` in feature 004 and `Category` in feature 005). The spec locks the indexing rules: `(userId, date)` for date-range queries, `(userId, accountId, date)` for per-account list with date order, `(userId, categoryId)` for category-aggregation queries (feature 008 budgets, feature 015 charts, feature 016 reports), and `(userId, transferGroupId)` for reconciling transfer pairs on edit and archive. The exact index definitions are plan-level; the spec locks the queries they MUST support.
- **FR-003**: Every query against `Transaction` (list, read-by-id, create, update, archive, unarchive, balance computation) MUST be scoped to the current session's `userId` at the route boundary. No route handler MAY accept a `userId` parameter from request input; the session is the sole source of truth for the requesting user's identity. Cross-user reads and writes (including reading another user's transaction by `id`, updating it, archiving it, unarchiving it, OR creating/editing a transaction whose `accountId` or `categoryId` references rows owned by another user) MUST be impossible to perform through the product surface and MUST collapse to a not-found-style envelope (FR-013).

#### Field rules

- **FR-004**: The `date` field MUST be a `DateTime` whose time component is normalized to 00:00:00 UTC of the user-submitted calendar day at the boundary. The user-facing form accepts a calendar date in the user's timezone; the boundary normalizes it to that calendar day's midnight UTC. Storage is UTC, display is calendar-day (no time component shown in the list). No upper or lower bound on the date in v1.
- **FR-005**: The `accountId` field MUST be required (a transaction always belongs to exactly one account on its primary leg; a transfer creates two transactions each with its own `accountId`). The referenced `Account` MUST be owned by the same `userId` AND MUST be non-archived at the moment of creation (a user cannot record a NEW transaction against an archived account — though existing transactions on a now-archived account retain their reference). For edit, the user may change `accountId` only to another account they own with the SAME currency (FR-007); reassign across currencies is rejected at the boundary.
- **FR-006**: The `categoryId` field MUST be optional (nullable) for `type=INCOME` and `type=EXPENSE` — a transaction MAY be uncategorized. The `categoryId` field MUST be null for `type=TRANSFER` rows (transfers are inherently uncategorized; they reclassify money rather than earn or spend it). When non-null, the referenced `Category` MUST be owned by the same `userId` AND MUST be of the matching `kind` (INCOME transaction → INCOME category; EXPENSE transaction → EXPENSE category) per FR-011.
- **FR-006a**: The category picker shown in the transaction create/edit form MUST exclude archived categories on NEW transactions. Existing transactions whose `categoryId` references a category that has since been archived MUST retain their reference and render in the list with the archived category's name in a muted treatment; the category picker on the edit sheet for such a transaction MUST include the currently-archived category as the active selection (so the user can leave it as-is) while still hiding other archived categories from the picker dropdown.
- **FR-007**: The `currency` field on a `Transaction` MUST be a three-letter uppercase ISO 4217 currency code AND MUST equal the `currency` of the referenced `Account` at all times. The boundary MUST reject any payload whose declared `currency` differs from the account's `currency`. In practice the create form does not let the user pick a currency directly — the currency is derived from the chosen account and persisted alongside the transaction for query/aggregation efficiency. This denormalization is intentional; the equality invariant is enforced at the boundary and (in the plan) by a CHECK constraint or equivalent.
- **FR-008**: The `amount` field MUST be a Decimal value (constitution Principle I — never `Float`/`Number`). It MUST be stored as Postgres `NUMERIC`. The boundary MUST enforce the signed-amount convention: for `type=INCOME` the stored `amount` MUST be strictly positive (`> 0`); for `type=EXPENSE` the stored `amount` MUST be strictly negative (`< 0`); for `type=TRANSFER` one leg MUST be strictly negative (the source) and the other leg MUST be strictly positive (the destination), with `abs(leg1.amount) === abs(leg2.amount)`. A zero amount on any transaction MUST be rejected at the boundary. The user-facing form accepts a positive magnitude; the boundary applies the sign per the chosen `type` (or rejects a tampered payload whose sign disagrees with the type).
- **FR-009**: The number of fractional digits accepted on the `amount` MUST match the convention for the chosen account's `currency` (e.g., 2 for USD/EUR/GBP, 0 for JPY/KRW, 3 for BHD/KWD/JOD). Submitting more fractional digits than the currency supports MUST be rejected at the boundary with an actionable error. This rule is inherited from feature 004 FR-006 and applied uniformly to transaction amounts.
- **FR-010**: The `payee` field, when non-null, MUST be a whitespace-trimmed string between 1 and 120 characters. The `notes` field, when non-null, MUST be a whitespace-trimmed string between 1 and 500 characters. An empty string (after trim) on either field MUST be normalized to null. Both fields are display-only; the system does not enforce uniqueness or pattern matching on either.
- **FR-011**: The `type` field MUST be a member of the closed enum `{INCOME, EXPENSE, TRANSFER}`. For `type=INCOME` and `type=EXPENSE`, when `categoryId` is non-null, the referenced category's `kind` MUST match the transaction's `type` (INCOME ↔ INCOME, EXPENSE ↔ EXPENSE). For `type=TRANSFER`, `categoryId` MUST be null (FR-006). The boundary MUST enforce both rules.
- **FR-012**: The `transferGroupId` field MUST be a system-generated string (the plan decides format — UUID, cuid, etc.). On create, the server MUST generate it; client-supplied values MUST be ignored. For `type=INCOME` and `type=EXPENSE`, `transferGroupId` MUST be null. For `type=TRANSFER`, BOTH legs MUST share the same non-null `transferGroupId`. The pairing rule is the canonical way the system reconciles transfer legs on edit and archive (FR-018, FR-024, FR-025).

#### Transfer atomicity

- **FR-013**: Every Transaction API endpoint (create, update, archive, unarchive) MUST validate input with Zod at the boundary before touching business logic or persistence, per constitution Principle III. Once validated, internal helpers MUST trust their typed inputs. Validation failures MUST return a `{ error: { code, message } }` envelope (FR-027); on failure, ZERO rows MUST be created or modified.
- **FR-014**: Creating a transfer MUST be an atomic operation. The system MUST create exactly two `Transaction` rows in a single database transaction (constitution Principle I — "Transfers between accounts are atomic: a single transaction creates two ledger entries in one DB transaction or it fails"). Both legs MUST be created together or neither persists. The two legs MUST share: the same `transferGroupId` (generated by the server), the same `date`, the same `currency`, the same `userId`, the same `notes` (or both null), `type = TRANSFER`. The source leg's `accountId` MUST be the source account's id and its `amount` MUST be strictly negative; the destination leg's `accountId` MUST be the destination account's id and its `amount` MUST be strictly positive; the absolute values of the two legs' amounts MUST be equal. Source and destination accounts MUST be different (`fromAccountId !== toAccountId`); a submission where they are equal MUST be rejected at the boundary (FR-013).
- **FR-015**: A transfer's two legs MUST share the same `currency`. In v1, that means the source account and the destination account MUST have the same `currency`. Cross-currency transfers MUST be rejected at the Zod boundary with an actionable error stating that cross-currency transfers are not supported in this version (FX conversion lands in roadmap feature 020). Zero rows MUST persist on a cross-currency attempt.
- **FR-016**: Updating a transfer MUST be an atomic operation. The system MUST fetch BOTH legs by the shared `transferGroupId`, apply the edit to both legs in a single database transaction, and either persist BOTH updates or NEITHER. The fields that MAY be updated on a transfer are: the source `accountId` (the From), the destination `accountId` (the To), the `date`, the `amount` magnitude, the `notes`. The source/destination pair after edit MUST continue to satisfy the FR-014 invariants (distinct accounts, same currency, valid `userId` ownership). A tampered API call that attempts to update only one leg of a transfer (via the single-leg update path with a `transferGroupId`-bearing row id) MUST be rejected at the boundary: any update to a row whose `transferGroupId` is non-null MUST route through the transfer-update path (FR-024).
- **FR-017**: Archiving a single-leg transaction (INCOME or EXPENSE) MUST set its `archivedAt` to the current server timestamp. The transaction MUST be excluded from the default list view (FR-019) and excluded from the per-account balance computation (FR-019a) while archived. Unarchiving clears `archivedAt`; the transaction returns to the default list and re-enters the balance computation. Both actions MUST be reversible an arbitrary number of times.
- **FR-018**: Archiving a transfer MUST archive BOTH legs atomically. The system MUST fetch both legs by the shared `transferGroupId` and set `archivedAt` to the same server timestamp on both rows in a single database transaction. Unarchive MUST mirror this: clearing `archivedAt` on both legs atomically. A tampered API call that attempts to archive only one leg (via the single-leg archive path on a row whose `transferGroupId` is non-null) MUST be rejected at the boundary; any archive/unarchive on a `transferGroupId`-bearing row MUST route through the transfer-archive path.

#### Balance computation

- **FR-019**: The default list view MUST exclude rows where `archivedAt` is non-null. A "Show archived" filter toggle (default off) MUST make archived rows visible alongside active rows with a muted treatment and an "Archived" badge. Archived rows MUST display the same fields as active rows; they are visually distinguished, not data-distinguished.
- **FR-019a**: An account's displayed balance MUST equal `account.startingBalance + Σ(transaction.amount for that account where archivedAt is null)`. This is the formula feature 004 FR-017 promised; feature 006 actually implements it. The computation MUST be centralized (no direct arithmetic on transaction amounts outside `lib/money/` — FR-028). Archived transactions MUST NOT contribute to the displayed balance; unarchiving MUST re-include them. The formula MUST hold byte-for-byte across every rendered surface that shows a balance (the accounts list, future dashboard widgets in feature 007, future reports in feature 016).

#### List + filter

- **FR-020**: The transactions list MUST be sorted by `date` descending (most recent first), with a stable secondary sort by `createdAt` descending (so two transactions on the same calendar day render in creation order, newest-created first). Sort is deterministic across reloads. User-defined sort and drag-to-reorder are OUT OF SCOPE.
- **FR-021**: The transactions page MUST expose TWO primary CTAs: "Add transaction" (opens the single-leg create sheet for INCOME or EXPENSE) and "Add transfer" (opens the transfer create sheet). The two sheets have different form shapes (FR-023, FR-024) and submit through different action paths. Combining them into one polymorphic form is explicitly out of scope; the UX clarity of two distinct affordances is the locked decision.
- **FR-022**: The account pickers in the create transaction sheet, the create transfer sheet, and the edit transaction/transfer sheets MUST list only non-archived accounts owned by the session's user. A new transaction cannot route to an archived account. An existing transaction whose account has since been archived continues to render correctly in the list (the account name is shown with a muted treatment), but the edit sheet's account picker excludes the archived account from the dropdown (preserving the existing selection if it is the currently-archived account, similar to FR-006a's handling of archived categories).
- **FR-023**: The single-leg create/edit sheet MUST present these fields: `account` (picker, required), `category` (picker, optional, kind-filtered by `type`), `date` (defaulted to today, required), `amount` (positive magnitude, required, > 0, currency-aware decimal places), `type` (INCOME | EXPENSE radio or segmented control, required, defaulted to EXPENSE on create), `payee` (optional, up to 120 chars), `notes` (optional, up to 500 chars). The form MUST surface the chosen account's currency next to the amount field so the user knows which currency they are entering.
- **FR-024**: The transfer create/edit sheet MUST present these fields: `From account` (picker, required), `To account` (picker, required, distinct from From), `Date` (defaulted to today, required), `Amount` (positive magnitude, required, > 0, currency-aware decimal places), `Notes` (optional, up to 500 chars). There MUST be no `Category` field, no `Type` field, no `Payee` field. The form MUST surface the source account's currency next to the amount field; selecting a `To account` of a different currency MUST surface an inline error (FR-015) before submit. The transfer-edit form operates on the pair as a unit; a single leg of a transfer MUST NOT be editable in isolation through any product surface.
- **FR-025**: Clicking a row in the list opens the corresponding edit sheet: single-leg row → single-leg edit sheet (FR-023 fields, pre-populated); transfer row (either leg) → transfer-edit sheet (FR-024 fields, pre-populated from both legs via the shared `transferGroupId`). The rendering of transfer rows in the list itself is plan-level — the list MAY show two paired rows (one for the source leg, one for the destination leg, both clearly marked as parts of the same transfer) OR a single combined row that expands on click — but the edit affordance MUST always open the transfer as a unit (FR-024).
- **FR-026**: The default date range on the transactions list MUST be the last 30 days (inclusive of today, the previous 30 calendar days). The user MUST be able to adjust the start and end of the range via a date-range picker. When the matching filter set contains more rows than fit in the initial render window, the list MUST provide a mechanism for the user to reach every matching row (paginated, infinite-scroll, or "load older" button — the exact mechanism is plan-level; the requirement locks reachability, not implementation).
- **FR-026a**: The list MUST support filtering by: date range (start + end), account (single-select with "All accounts" option), category (single-select with "All categories" and "(uncategorized)" options, surfaced through the `<CategoryPicker>` primitive from feature 005), type (single-select with "All types", "Income", "Expense", "Transfer" options), and archive state ("Show archived" toggle, default off). All filter state SHOULD be URL-encoded so the filtered view is shareable and reload-stable; the exact URL parameter shape is plan-level.

#### Error & validation contract

- **FR-027**: All Transaction API endpoints MUST conform to the constitution's response envelope: success returns `{ data }`, failure returns `{ error: { code, message } }`. HTTP status reflects outcome (2xx success, 4xx client error, 5xx server error). On a validation rejection, the response MUST include enough field-scoped detail for the form to highlight the specific offending field; the entered values for still-valid fields MUST be preserved on the client so the user can correct the one offending field and re-submit (US6 scenario 11).

#### Money math boundary

- **FR-028**: Every monetary write introduced by this feature MUST go through `lib/money/` helpers. The helpers MUST include at minimum: a Decimal arithmetic primitive (for sign assignment, sum, abs, comparison), a transfer-pair invariant check (both legs share `transferGroupId`, same currency, same date, same userId, inverse amounts), and a balance-computation helper that produces `startingBalance + Σ(non-archived transactions for an account)` from a Decimal-typed input set. No file outside `lib/money/` MUST perform direct arithmetic on transaction amounts (constitution Principle I and Conventions: "Money helpers: all monetary operations go through `lib/money/`"). Display formatting continues to happen at the UI edge via the `<Money>` primitive from feature 005.
- **FR-029**: When the user has no accounts (active or archived), the transactions page MUST render an empty state explaining that an account must exist first, with a clear pointer to `/dashboard/accounts`. The "Add transaction" and "Add transfer" CTAs MUST be disabled in this state (or replaced with the account-onboarding CTA). This prevents the awkward state of a form whose required `account` picker has nothing to pick.

#### Quality

- **FR-030**: The transactions list, the create transaction sheet, the create transfer sheet, the edit sheets, and the filter controls MUST all be operable by keyboard alone. Form fields MUST have associated labels; error messages MUST be programmatically associated with their fields. Color MUST NOT be the sole carrier of meaning (e.g., the negative-amount treatment on EXPENSE rows MUST also include the `-` sign or the currency-with-sign formatting from the `<Money>` primitive; the "Archived" badge MUST be a text label, not a color-only indicator). Numeric amounts in the list MUST use tabular numerals so columns of money values align column-wise (inherited from feature 005's `<Money>` primitive contract).
- **FR-031**: All new code MUST satisfy TypeScript strict mode with no use of `any` (constitution Principle II). All boundary schemas MUST be defined with Zod (constitution Principle II and III). No file outside the transaction module's boundary MUST perform direct validation on transaction input — the Zod schemas defined at the boundary are the single source of truth.
- **FR-032**: Unit tests MUST cover the money-correctness paths introduced by this feature: the transfer-pair invariant (both legs created atomically; both legs share `transferGroupId`; both legs match in currency, date, userId; both legs are inverses in amount per the signed-amount convention), the sign-must-match-type rule (INCOME → amount > 0; EXPENSE → amount < 0; TRANSFER → one negative + one positive leg), the currency-must-match-account rule, the cross-currency-transfer rejection, the zero-amount rejection, the currency-aware decimal-places rule (inherited from feature 004 and applied to transaction amounts), and the balance-computation formula (`startingBalance + Σ(non-archived transactions where accountId matches)`). These are the constitution Principle IV deliverables for this feature.
- **FR-033**: A Playwright E2E MUST cover the constitution-mandated "create transaction, transfer between accounts" path: sign in, create an EXPENSE transaction, verify the source account's balance decreased by the expected amount; create an INCOME transaction, verify the source account's balance increased by the expected amount; create a TRANSFER between two same-currency accounts, verify both accounts' balances moved correctly AND verify the transfer was persisted as two rows sharing a `transferGroupId` with inverse amounts. The E2E MUST assert the atomicity invariant on the transfer (BOTH legs present, OR neither present — never just one).
- **FR-034**: This feature MUST NOT introduce auto-categorization rules (feature 012), CSV import (feature 010) or export (feature 014), recurring transactions (feature 011), multi-currency / FX conversion on transfers (feature 020), tags on transactions (feature 013), attachments or receipts (feature 018), charts (feature 015), aggregated reports by category / payee / month (feature 016), an audit log (feature 021), or real-time updates (no WebSocket / SSE in v1; the list refreshes on user action only). Each is explicitly deferred to a later roadmap feature.

### Key Entities

- **Transaction** — A single recorded money event belonging to one user and one account. Identified by a stable unique identifier. Attributes:
  - `userId` — the owning user's identifier; the row is unreachable to any other user; cascade-deletes with the user.
  - `accountId` — references exactly one `Account` (owned by the same user). Restrict-on-delete at the FK layer; account hard-delete is already prevented at the UI layer (feature 004 FR-008).
  - `categoryId` — nullable. References at most one `Category` (owned by the same user, of matching `kind` per FR-011). For `type=TRANSFER` rows, MUST be null. Restrict-on-delete at the FK layer; category hard-delete is already prevented at the UI layer (feature 005 FR-010).
  - `date` — a `DateTime` normalized to 00:00:00 UTC of the user-submitted calendar day. Represents the day the money moved, not the day the user recorded it (`createdAt` carries that).
  - `amount` — a signed Decimal value in the account's currency. Positive for INCOME, negative for EXPENSE, signed-pair for TRANSFER (source leg negative, destination leg positive). Stored as Postgres `NUMERIC` (never float). Magnitude strictly > 0.
  - `currency` — the three-letter ISO 4217 code. ALWAYS equal to the referenced account's `currency` at the boundary (FR-007). Denormalized for query efficiency.
  - `type` — one of `INCOME | EXPENSE | TRANSFER`. The sign of `amount` MUST be consistent with `type` per FR-008.
  - `payee` — optional string (1–120 chars after trim), nullable. Display-only; not constrained to a closed set or pattern.
  - `notes` — optional string (1–500 chars after trim), nullable. Display-only.
  - `transferGroupId` — nullable string. Null for INCOME and EXPENSE rows. Non-null for TRANSFER rows; both legs of a transfer share the same value. The pairing key the system uses to atomically reconcile, edit, archive, and unarchive a transfer.
  - `archivedAt` — nullable timestamp. Non-null means the transaction is archived (hidden from the default list and the balance computation, retained for referential integrity). Reversible. Archive on a transfer leg cascades atomically to the other leg via shared `transferGroupId` (FR-018).
  - `createdAt`, `updatedAt` — standard timestamps, stored UTC, full precision.
  - Relationship: every `Transaction` belongs to exactly one `User` and one `Account`. Every `Transaction` MAY reference one `Category` (nullable). Two `Transaction` rows sharing the same non-null `transferGroupId` form one transfer; they share `userId`, `currency`, `date`, `notes`, and `type=TRANSFER`, and their amounts are inverses (one negative, one positive, equal magnitude).
- **(Future-only references — NOT created in this feature)** — `Budget` (feature 008) will aggregate `Transaction.amount` over a date range and a `categoryId`; the dashboard widgets in feature 007 will aggregate `Transaction.amount` over `userId` and `accountId` for net-worth and over `userId` and date range for cash flow; feature 015 (Charts) and feature 016 (Reports) will read transactions for visualizations and reports. This feature does NOT add any of those entities and MUST NOT presume their shape beyond "they will read transactions, scoped by `userId`, filtered by `archivedAt = null`, and will rely on the balance-computation formula (FR-019a) being authoritative."

## Success Criteria *(mandatory)*

- **SC-005**: Archiving a transfer leg archives BOTH legs in a single database transaction (both `archivedAt` set to the same timestamp) in 100% of attempts. Unarchiving symmetrically clears BOTH legs' `archivedAt`. Archiving one leg without the other is impossible through any product surface.

## Assumptions

- Feature 003's auth boundary is on `main`: every `/dashboard/*` route requires a session; the session is the canonical source of the requesting user's identity (`userId`); cross-user access is prevented at the route boundary, not in helpers. This feature builds on that boundary without re-implementing it.
- Feature 004's `Account` model is on `main`: it carries `userId`, `currency`, `startingBalance`, and `archivedAt`; it uses the side-sheet edit pattern; account hard-delete is not exposed in the UI. This feature wires its `accountId` FK against this model with restrict-on-delete semantics at the data layer (because an account with transactions cannot be hard-deleted by definition).
- Feature 005's `Category` model is on `main`: it carries `userId`, `kind` (INCOME | EXPENSE), `parentId` for single-level hierarchy, and `archivedAt`. It uses the side-sheet edit pattern and ships a reusable `<CategoryPicker>` primitive. This feature wires its `categoryId` FK against this model with restrict-on-delete semantics at the data layer, makes the FK nullable (uncategorized transactions are allowed for INCOME and EXPENSE; forced for TRANSFER), and consumes the `<CategoryPicker>` primitive in the transaction create/edit sheet.
- Feature 005 also shipped the `<Money>` display primitive and the `<EmptyState>` primitive on `main`. This feature uses both: `<Money>` for every amount rendered in the list, the sheets, and any balance display this feature touches; `<EmptyState>` for the no-transactions-yet view and the no-accounts-yet view (FR-029).
- The `lib/money/` module was established in feature 004 with at minimum a Decimal value abstraction and a currency-aware formatter. This feature extends it with the additional helpers required for transactions: signed-amount construction, sign-must-match-type validation, the transfer-pair invariant check, and the balance-computation helper (FR-028). The exact helper signatures are plan-level; the spec locks the boundary rule (no arithmetic on transaction amounts outside `lib/money/`) and the new test coverage (FR-032).
- The transactions module follows the lib-module pattern feature 004 and feature 005 established (`actions.ts`, `queries.ts`, `schemas.ts`, `serialize.ts`, `errors.ts`, `index.ts`). The exact file inventory is plan-level; the spec locks the conventions (Zod schemas at the boundary, internal helpers trust typed inputs, error envelope on all responses).
- The default date range on the transactions list is "the last 30 days" (FR-026). Other reasonable defaults exist ("current calendar month", "last 7 days"); the spec locks 30 days as a balance between recency (most users want to see what they just recorded) and reach (users who recorded transactions a few weeks ago can still see them without changing the filter). A future feature MAY add a user-configurable default; v1 does not.
- The transactions list's rendering of transfer rows (one combined row vs. two paired rows) is plan-level (FR-025). The spec only locks the edit affordance: the transfer is always edited as a unit, never one leg in isolation.
- URL-encoded filter state (FR-026a) is the recommended pattern; the spec locks "filter state SHOULD be URL-encoded" but leaves the exact parameter names to the plan, in line with feature 009 (Search & filter) building on this contract.
- The Playwright E2E surface for this feature (FR-033) is the constitution-mandated "create transaction, transfer between accounts" path. The exact test fixture data, the test file location, and the assertion mechanism are plan-level; the spec locks the coverage (all three flows: INCOME, EXPENSE, TRANSFER) and the atomicity assertion.
- The money-reviewer subagent will audit this PR per the constitution Subagent Workflow. Their `PASS` criteria are codified in SC-017. The spec does not pre-empt their judgment; it commits to a surface the reviewer can audit cleanly.
- The roadmap-numbered feature 006 (Transactions + Transfers) lives in spec-dir `007-transactions` because the branded-UI polish chore consumed spec-dir slot `005-branded-ui-polish` in May 2026. The roadmap number and spec-dir number diverge by one slot starting with feature 005 (Categories, in spec-dir `006-categories`) and continuing here.

## Out of Scope

- **Auto-categorization rules** — pattern-match a payee or description to a category and apply automatically. Lands in roadmap feature 012. Categorization in this feature is purely manual.
- **CSV import** — bulk-loading transactions from a bank file. Lands in roadmap feature 010.
- **CSV export** — exporting transactions to a CSV file. Lands in roadmap feature 014.
- **Recurring transactions** — cron-style or "next due" scheduler for monthly rent, salary, subscriptions. Lands in roadmap feature 011.
- **Multi-currency transfers / FX conversion** — transferring between accounts of different currencies with an exchange rate. Lands in roadmap feature 020. Same-currency transfers only in v1 (FR-015).
- **Tags on transactions** — free-form labels orthogonal to categories. Lands in roadmap feature 013. Categories carry the structured classification in v1.
- **Attachments / receipts** — uploading a photo of a receipt against a transaction. Lands in roadmap feature 018.
- **Charts** — visualizations of spending over time, by category, by account. Recharts lands in roadmap feature 015. This feature ships a date-range-filtered list, not charts.
- **Reports** — aggregated views by category, payee, month, with drill-down. Lands in roadmap feature 016. This feature ships a list, not aggregations.
- **Audit log** — who changed which transaction when, with diff. Lands in roadmap feature 021.
- **Real-time updates** — no WebSocket / SSE in v1; the list refreshes on user action only.
- **Hard delete** of transactions or transfers — only archive is exposed (per the 2026-05-17 clarification). A future feature MAY introduce hard delete with safeguards (e.g., only transactions younger than N days, only never-referenced rows); this feature does not.
- **Pending vs. cleared state** — every transaction in v1 is implicitly cleared. A future feature MAY introduce a "pending" flag (e.g., for a transaction the bank has not yet posted); v1 does not.
- **Split transactions** — one expense allocated across multiple categories. A future feature MAY introduce this; v1 requires a single `categoryId` per row.
- **Per-transaction currency override** — a transaction's currency MUST equal its account's currency (FR-007). Recording a foreign-currency expense against a USD account (with implicit FX) is out of scope; the user creates a separate account in that currency.
- **Cross-user transaction sharing** — Abacus has no shared workspaces per feature 003. A transaction belongs to exactly one user and is never visible to another user.
- **Bank syncing / Plaid / open banking integrations** — deliberately deferred (see roadmap "Deliberately deferred").
- **Aggregated cross-currency totals** — multi-currency net-worth via FX. Lands together in roadmap feature 020 alongside cross-currency transfers.
- **Per-account detail page with embedded transactions list** — a future feature MAY introduce `/dashboard/accounts/[id]` with the per-account transactions view. v1 ships the transactions list at the top-level `/dashboard/transactions` route with an account filter; that is the canonical surface.

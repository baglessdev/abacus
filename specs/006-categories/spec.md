# Feature Specification: Categories

**Feature Branch**: `006-categories`

**Created**: 2026-05-17

**Status**: Draft

**Input**: User description: "Categories now" (kicked off via `/speckit-specify` after the branded-UI chore merged)

> **Numbering note**: This is the roadmap-numbered feature **005 — Categories** (Tier 1). The spec directory is `006-categories` because the branded-UI polish chore consumed spec-dir slot `005-branded-ui-polish`. The roadmap entry and this spec describe the same feature; the directory number tracks order-of-spec-authored, not the roadmap slot.

## Why

Categories are the user's classification system for the money flowing through their accounts: "Groceries", "Rent", "Salary", "Pet Care". Without them, every transaction is an undifferentiated dollar amount and the downstream features that make Abacus useful — Budgets (cap spending per category), Reports (aggregate by category), Auto-categorization rules (match a payee to a category) — have nothing to operate on. Feature 004 established the data-scoping convention (every domain row is owned by a `userId`; every query filters by the session's user) on a money-touching model; feature 005 is the second domain entity to exercise that convention, and the first one whose shape — the parent/child hierarchy — directly locks the schema that features 006, 008, 012, and 016 will plug into. Getting the category model and its management UX right now means none of those later features need a Category-schema migration to work, and the user sees a populated, brand-coherent Categories surface from the moment they finish signup.

## Clarifications

### Session 2026-05-17

- Q: Hierarchy depth — single level (one parent only, no grandchildren) or unbounded depth? → A: Single level. A category is either a top-level category (`parentId` is null) or a child of a top-level category (`parentId` references a top-level category). A child category MUST NOT have its own children. This is the convention in YNAB, Copilot Money, and Monarch; it is sufficient for personal finance (a "Pets" → "Pet Food" / "Vet" split is the common shape, and deeper nesting hurts picker UX more than it helps). It also removes the need for cycle detection and depth-limiting in validation. The data model still uses a self-referential `parentId` so a future feature could lift the restriction without a destructive migration; this feature simply rejects any payload with `parentId` referencing a category that itself has a non-null `parentId`.
- Q: Route location — top-level `/dashboard/categories` or nested under settings as `/dashboard/settings/categories`? → A: Top-level `/dashboard/categories`. Categories are referenced by Transactions and Budgets multiple times a day; the user will visit this page often to rename, add, or archive a category as their financial life evolves. Top-level placement in the sidebar's MANAGE group (alongside Accounts) matches that frequency and matches the discoverability bar set by feature 004. Settings is reserved for true administrative configuration (profile, primary currency, password) per the roadmap's feature 017.
- Q: Default seed on first signup — opinionated default set or empty by default? → A: Opinionated default set, seeded inside the signup transaction so it is atomic with user creation. The seed is exactly eleven categories: seven top-level EXPENSE categories ("Food", "Housing", "Transport", "Utilities", "Entertainment", "Health", "Other Expenses"), two top-level INCOME categories ("Salary", "Other Income"), and two child categories under Food to demonstrate the hierarchy ("Groceries" and "Restaurants", both EXPENSE, both with `parentId` referencing "Food"). Each seeded category has a stable default `color` token and `icon` identifier from the curated set (FR-007 and FR-008). The seed exists so a newly-signed-up user lands on a non-empty Categories page and so feature 006's first transaction-creation flow has something to pick from. If the seed write fails for any reason, the signup transaction MUST roll back; a signed-up user with no categories is never a reachable state.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - A new user lands on a non-empty Categories page (Priority: P1)

As a user who just finished signup, I open the Categories page (or arrive there via the sidebar's MANAGE group) and immediately see a populated list of sensible default categories — both my expense categories and my income categories — grouped and laid out clearly enough that I can start using them right away on my first transaction, even if I never customize them.

**Why this priority**: This is the on-ramp from "signup complete" to "the app is useful". An empty Categories page on a fresh account would force every new user to invent the same 7–10 categories from scratch before they could record their first transaction. The default seed compresses that ten minutes of friction to zero. P1 because it is the difference between a usable first session and a stalled one, and because the seed must be in place before US2–US5 are meaningful to demo.

**Independent Test**: Sign up a brand-new user. Without performing any other action, navigate to `/dashboard/categories`. The page renders a populated list — not an empty state — containing the eleven seeded categories from the clarifications: seven top-level EXPENSE categories ("Food", "Housing", "Transport", "Utilities", "Entertainment", "Health", "Other Expenses"), two top-level INCOME categories ("Salary", "Other Income"), and two child EXPENSE categories under Food ("Groceries", "Restaurants"). The hierarchy is visible (children render visually under their parent). The two `kind`s are grouped or otherwise visually distinguished so the user can scan income separately from expense.

**Acceptance Scenarios**:

1. **Given** a brand-new user has just completed signup, **When** they navigate to `/dashboard/categories` for the first time, **Then** the page renders a populated category list with the eleven seeded categories from the clarifications. No empty state is rendered.
2. **Given** the user is viewing the categories list, **When** the list renders, **Then** EXPENSE categories and INCOME categories are visually grouped (separate sections or a clear visual band) so the user can read them as two independent classifications.
3. **Given** the user is viewing the categories list, **When** the list renders, **Then** the two child categories ("Groceries", "Restaurants") render visually under their parent ("Food") — for example, indented or grouped — so the parent/child relationship is unambiguous.
4. **Given** the user reloads the page, **When** the list re-renders, **Then** the same eleven seeded categories are still present in the same shape (the seed persisted, scoped to this user).
5. **Given** a SECOND brand-new user signs up, **When** they open `/dashboard/categories`, **Then** they see THEIR OWN eleven seeded categories — not the first user's. Editing one user's seeded "Food" does not affect the other user's "Food" (the rows are independent, owned by different `userId`s).

### User Story 2 - Create a new category (Priority: P1)

As a user with the default seeded categories already in place, I open the create form, fill in a name (e.g., "Pets"), pick `EXPENSE` as the kind, pick a color from a curated palette, pick an icon from a curated set, optionally pick a parent (for a child category), submit, and immediately see the new category in the list under the right kind and (if applicable) the right parent.

**Why this priority**: The defaults won't cover every user's life. "Pets", "Side hustle income", "Childcare", "Subscriptions", "Charity" — every user has a few categories that are specific to them. Without a create flow, the app is locked to the seed forever. P1 because the create flow is the path by which Categories becomes a tool the user actually owns, and the picker that future features 006 and 008 will consume needs to surface every category the user has — seeded or custom — uniformly.

**Independent Test**: From the categories page, activate the primary "Add category" CTA. A side sheet opens in "create" mode. Fill `name = "Pets"`, `kind = EXPENSE`, pick a color, pick an icon, leave parent unset, submit. The sheet closes; the new "Pets" category appears in the EXPENSE section of the list. Open the create CTA again. Fill `name = "Vet"`, `kind = EXPENSE`, pick a color, pick an icon, set `parent = "Pets"`, submit. The sheet closes; "Vet" appears in the list visually nested under "Pets".

**Acceptance Scenarios**:

1. **Given** the categories page is open, **When** the user activates the primary "Add category" CTA, **Then** a side sheet opens in "create" mode with empty fields (name, kind, color default, icon default, parent unset).
2. **Given** the create sheet is open, **When** the user submits a valid top-level category (name `Pets`, kind `EXPENSE`, color from the palette, icon from the curated set, no parent), **Then** the category is persisted, the sheet closes, and the new row appears in the EXPENSE section of the list with its chosen name, color, and icon visible.
3. **Given** the create sheet is open, **When** the user submits a valid child category (name `Vet`, kind `EXPENSE`, color, icon, parent set to an existing top-level EXPENSE category they own), **Then** the category is persisted, the sheet closes, and the new row appears nested under its parent in the list.
4. **Given** the create sheet is open, **When** the user selects a parent for the new category, **Then** the kind field is forced to match the parent's kind (a child of an EXPENSE parent MUST itself be EXPENSE; a child of an INCOME parent MUST itself be INCOME). The form surfaces this constraint clearly — e.g., the kind control becomes read-only and shows the inherited value once a parent is selected.
5. **Given** the user has just created a new category, **When** they reload the page, **Then** the category is still present (the data persisted, scoped to this user).
6. **Given** a SECOND user attempts to create a child category whose `parentId` references a category owned by the FIRST user, **Then** the operation MUST fail with a not-found-style error envelope (FR-013); the parent picker MUST NOT surface another user's categories in any session.

### User Story 3 - Edit, archive, and unarchive a category (Priority: P1)

As a user with at least one category already (seeded or self-created), I can open an existing category from the list, change its name, change its color, change its icon, change its kind (if it has no children), reassign its parent (or unset it), archive it when I no longer use it, and later unarchive it if I made a mistake. The list updates immediately to reflect each change.

**Why this priority**: Personal finance is not static. The user may decide "Entertainment" is too broad and rename it to "Subscriptions", or that "Other Income" was redundant and want to archive it, or that "Pet Food" should move from being a child of "Food" to being a child of "Pets" once they create the latter. Without edit and archive, the category set ossifies at whatever the seed plus the first creation pass produced. P1 because the create-only flow alone is not viable: users would have no way to correct typos or evolve their classification, and there is no delete affordance (archive is the only reversible non-destructive escape hatch — the same pattern feature 004 established for accounts and the same rule the downstream features will depend on for referential integrity).

**Independent Test**: With a user who already owns at least one category, click an existing category row. A side sheet opens in "edit" mode pre-populated with the current values. Change the name, save: the row updates. Reopen the row, change the color and icon, save: the row updates. Reopen the row, archive: the row leaves the default list view. Activate "Show archived": the row reappears with an "Archived" badge. Reopen the row, unarchive: the row returns to the default list view without the badge.

**Acceptance Scenarios**:

1. **Given** the user has at least one category, **When** they click that category in the list, **Then** a side sheet opens in "edit" mode pre-populated with the category's current name, kind, color, icon, parent (if any), and archive state.
2. **Given** the edit sheet is open on a top-level category, **When** the user changes the name to a valid new value and submits, **Then** the category's name updates, the sheet closes, and the list shows the new name in the row. Any child categories continue to render correctly under it.
3. **Given** the edit sheet is open, **When** the user changes the color and icon to valid values from the curated set and submits, **Then** the color and icon update on the row in the list. The change is visible without requiring a page reload.
4. **Given** the edit sheet is open on a category that has NO children, **When** the user changes the kind (e.g., EXPENSE → INCOME) and submits, **Then** the kind updates and the row moves to the corresponding kind's section of the list.
5. **Given** the edit sheet is open on a category that HAS children, **When** the user attempts to change the kind, **Then** the kind control is read-only (or the submission is rejected at the boundary with an actionable error) because changing a parent's kind would orphan its children whose kind must match (FR-005). The user is told to either move or archive the children first.
6. **Given** the edit sheet is open, **When** the user changes the parent (assigning, reassigning, or clearing) to a valid value owned by the same user with a matching kind and itself a top-level category, **Then** the parent updates and the row's position in the list reflects the new parent (nested under the new parent or promoted to top-level if the parent was cleared).
7. **Given** the edit sheet is open, **When** the user activates "Archive" and confirms, **Then** `archivedAt` is set, the sheet closes, and the row disappears from the default list view. If the category had children, the children remain visible at their own level — they are NOT cascade-archived (FR-010).
8. **Given** the user has at least one archived category, **When** they enable the "Show archived" toggle, **Then** the archived row appears in the list with an "Archived" badge and a muted treatment. Active rows remain visible above (or in their usual position).
9. **Given** an archived category's edit sheet is open, **When** the user activates "Unarchive", **Then** `archivedAt` is cleared, the row returns to the default list view without the badge, and on the next edit-sheet open all fields are editable per the normal rules.
10. **Given** a user owns categories A and B, and a SECOND user owns category C, **When** the first user submits an edit payload that references C by id, **Then** the operation MUST fail with a not-found-style error envelope (FR-013); the second user's category MUST NOT be reachable or modifiable through any product surface to the first user.

### User Story 4 - The category picker is consumable by downstream features (Priority: P2)

As a future user filling out a transaction form (feature 006) or a budget form (feature 008), I need to pick a category from a control that lists every category I own, grouped by kind, with the hierarchy visible, with archived categories hidden by default, and that returns the picked category's stable identifier to the consuming form. The same control must be reusable across those future features without forking the implementation.

**Why this priority**: Categories exist to be picked. The category-list page (US1–US3) is the management surface; the picker is the consumption surface. Locking the picker contract in this feature means feature 006 (Transactions) and feature 008 (Budgets) can plug it in without renegotiating the API. P2 because the management surface is independently testable and demoable without the picker — but the picker contract has to be defined in this feature so downstream features can wire to it without a follow-up rework.

**Independent Test**: A standalone demo surface (e.g., a small test page or a Storybook-style isolation) renders the category picker primitive. The picker lists every non-archived category the session's user owns, grouped by kind (INCOME vs EXPENSE), with children visually nested under their parent. Selecting a category fires a callback or updates form state with the chosen category's stable identifier. Archived categories are not shown in the default rendering. The same picker, mounted in a different consumer with different surrounding form fields, behaves identically.

**Acceptance Scenarios**:

1. **Given** the category picker is mounted in any consumer, **When** it renders, **Then** it lists every category owned by the session's user where `archivedAt` is null, grouped by kind (INCOME / EXPENSE), with children visually nested under their parent.
2. **Given** the picker is rendered, **When** the user selects a category, **Then** the consumer receives the selected category's stable identifier through a defined interface (the exact shape is plan-level; this spec only locks that the consumer is told WHICH category was picked, by id).
3. **Given** the picker is rendered for user A, **When** user B has categories, **Then** user A's picker MUST NOT surface any of user B's categories under any circumstance (data-scoping convention; FR-003).
4. **Given** a category has been archived (US3 scenario 7), **When** the picker re-renders or a new picker is mounted, **Then** the archived category is not listed in the default view. A future feature MAY introduce an "include archived" mode on the picker, but the DEFAULT MUST hide archived rows.
5. **Given** a category is a parent with children, **When** the picker renders, **Then** both the parent AND its children are selectable as independent picks (a transaction may be categorized as "Food" generically or as "Groceries" specifically; both are valid targets).
6. **Given** the picker is mounted in feature 006's transaction form (future) AND feature 008's budget form (future), **When** each one renders, **Then** both consumers display the same list shape and respond to selection the same way — there is exactly one picker implementation, not two parallel ones.

### User Story 5 - Validation surfaces actionable errors (Priority: P2)

As a user filling out the create or edit form for a category, if I submit invalid input — a blank name, a name longer than the maximum, a kind not in the enum, a color not in the curated palette, an icon not in the curated set, a parent that does not exist or belongs to another user, a parent whose kind does not match, a parent that is itself a child (would create a grandchild), the category as its own parent — the form rejects the submission, shows me a clear field-scoped error explaining what is wrong, and keeps my other entered values intact so I can fix the one offending field and try again.

**Why this priority**: Without per-field validation feedback, the form is opaque on the very first interaction. P2 because US1–US3 functionally pass without polished validation messages, but the UX gap is jarring on a surface the user will touch repeatedly over months. Several of the validations (parent-kind match, no-grandchildren, no-self-parent, parent-belongs-to-same-user) are also the load-bearing data-integrity rules that protect downstream features (Transactions, Budgets) from referential surprises, so they must be enforced at the boundary, not only suggested in the UI.

**Independent Test**: Submitting the create or edit form with each of the following invalid inputs surfaces a clear, field-scoped error and does NOT persist any change: blank name, whitespace-only name, name longer than 80 chars, kind missing, kind not in `{INCOME, EXPENSE}`, color missing or not in the curated palette, icon missing or not in the curated set, `parentId` referencing a non-existent id, `parentId` referencing a category owned by another user, `parentId` referencing a category whose kind differs from this category's kind, `parentId` referencing a category that itself has a non-null `parentId` (would create a grandchild), `parentId` equal to this category's own id (self-parent on edit).

**Acceptance Scenarios**:

1. **Given** the create form is open, **When** the user submits with a blank or whitespace-only name, **Then** the form shows an actionable name-field error and does not create the category.
2. **Given** the create form is open, **When** the user submits with a name longer than 80 characters (FR-004), **Then** the form shows an actionable length-error and does not create the category.
3. **Given** the create form is open, **When** the user submits with a kind not in `{INCOME, EXPENSE}` (or the kind field is missing), **Then** the form shows an actionable kind-field error and does not create the category.
4. **Given** the create form is open, **When** the user submits with a color or icon that is not in the curated allow-list (FR-007 and FR-008), **Then** the form shows an actionable error scoped to whichever field is invalid and does not create the category.
5. **Given** the create form is open, **When** the user submits with a `parentId` that does not exist, OR exists but belongs to a different user, **Then** the form shows a not-found-style error and does not create the category. The two cases MUST NOT be distinguishable in the response (FR-013).
6. **Given** the create form is open, **When** the user submits with a `parentId` whose category has a kind different from the submitted kind, **Then** the form shows an actionable error explaining that parent and child must share a kind, and does not create the category.
7. **Given** the create form is open, **When** the user submits with a `parentId` referencing a category that itself has a non-null `parentId` (i.e., a child trying to be a grandparent), **Then** the form shows an actionable error explaining the single-level depth rule, and does not create the category (FR-006).
8. **Given** the edit form is open on category X, **When** the user submits with `parentId` equal to X's own id, **Then** the form shows an actionable error explaining a category cannot be its own parent, and does not update the category.
9. **Given** the form has just been rejected on the server, **When** the form re-renders, **Then** the entered values are preserved for the still-valid fields, the offending field is focused, and the user can correct it and re-submit.
10. **Given** the user is editing an existing category, **When** any of the validation rules above fail on submit, **Then** the existing persisted category is NOT modified (atomic save: all-or-nothing).

### Edge Cases

- **A user attempts to view, edit, or archive a category that belongs to another user** (e.g., by guessing or tampering with an `id` in a request) — the operation MUST fail with a not-found-style error envelope (FR-013). The system MUST NOT distinguish "this category belongs to another user" from "this category does not exist" in any user-visible response, log entry, or response-timing channel.
- **A user creates two categories with the exact same name** (e.g., two "Other") — allowed. Names are display labels, not identifiers; the system MUST NOT enforce per-user name uniqueness. The list MAY render a discreet hint (e.g., the parent name as a secondary line, or the creation date) but is not required to. This is consistent with the analogous decision in feature 004 (FR-004 / edge cases) for `Account` names.
- **A user submits a name with surrounding whitespace** (e.g., `"  Groceries  "`) — the form trims whitespace at the boundary, then validates the trimmed value against the 1–80-character rule (FR-004). The persisted name is the trimmed form.
- **A user attempts to "delete" a category** — there is no hard-delete affordance in this feature. The only destructive-feeling action is "Archive", which is reversible (FR-010). This protects the referential integrity that feature 006's transactions will depend on once they reference categories by id.
- **A user archives a parent category that has children** — allowed. The parent is archived; its children are NOT cascade-archived (FR-010). Children whose parent is archived continue to render correctly in the default list (their parent is hidden by the default filter; the children appear at their normal position with a clear visual indication that their parent is archived, OR they continue to render under the archived parent only when "Show archived" is on — the exact UX detail is plan-level, but the data behavior is locked: no cascade).
- **A user archives all their categories** — allowed. The default list view is empty. The empty state from US1 is rendered (with the same illustration + primary CTA), since the page is functionally back to the same shape as a brand-new user who has not yet been seeded — except the user can also toggle "Show archived" to see and unarchive their old categories. This is consistent with feature 004's "user archives their only account" edge case.
- **A user tries to set their newly-created category as a child of itself in the create form** — impossible by construction (the category does not exist yet, so its id cannot be in the parent picker). The case is only meaningful in the edit form; FR-006 + US5 scenario 8 cover it.
- **A user tries to set category A as a child of category B, where B is currently a child of A** — impossible by construction in this feature, because B can only be a child of A if B's `parentId` is A and A has no `parentId`; that already makes A a top-level category. The only way A could then become a child of B is if B (a child) became a parent of A (a top-level) — which would require B's children to become A's grandchildren and violate FR-006. The rejection happens at the FR-006 boundary; no separate cycle-detection algorithm is required given the single-level rule.
- **A user submits a kind change on a category that has children** — rejected at the boundary (FR-005 + US3 scenario 5). The user is told to move or archive the children first. This is the data-integrity rule that keeps "parent and child share a kind" true across edits.
- **A user submits a parent change that would orphan the category's own children** (e.g., changing a parent's kind by way of changing its parent) — impossible by construction: a category with children is a top-level category, and a top-level category has `parentId = null`. Setting a parent on a category that has children would make IT a child, which (combined with FR-006's no-grandchildren rule) is rejected at the boundary.
- **A brand-new user's signup fails partway through seeding** — the entire signup transaction MUST roll back. A user row with no categories MUST never be a reachable state (FR-012). The seed write is part of the same DB transaction as the user-row creation, not a follow-up.
- **A user with no session attempts to reach `/dashboard/categories`** — covered by feature 003's auth boundary; the request is redirected to `/login` with the path preserved. This feature inherits that behavior and does not re-implement it.
- **A user with a session attempts to operate on a category by sending a tampered request body that includes a `userId` field** — the request body's `userId` (if present) is ignored. The session is the canonical source of identity (FR-003). Tampered payloads succeed only if they are valid for the SESSION's user, not for the user the payload tried to spoof.
- **A user has many categories** (dozens of top-level, each with several children) — the list remains usable. There is no enforced hard limit in v1. The list is designed for a few dozen top-level categories with single-level children, which is the realistic personal-finance ceiling.

## Requirements *(mandatory)*

### Functional Requirements

#### Data model

- **FR-001**: The system MUST introduce a `Category` domain model. At minimum the model MUST include a unique identifier, the owning user's identifier (foreign key to `User` with cascade-on-delete semantics), a user-visible name, a nullable self-referential `parentId` for hierarchy, a `kind` (closed enum: `INCOME | EXPENSE`), a `color` token, an `icon` identifier, an optional `archivedAt` timestamp, and creation/updated-at timestamps. The model MUST land via a generated Prisma migration (no `db push` against committed code, per the constitution).
- **FR-002**: Every `Category` row MUST carry a `userId` foreign key referencing `User.id` with cascade-on-delete semantics, indexed by `userId` (and indexed by `(userId, archivedAt)` to support the default-list query), so that removing a user removes their categories in a single referential step. This is the second domain entity to exercise feature 003's FR-025 data-scoping convention (the first was `Account`).
- **FR-003**: Every query against `Category` (list, read-by-id, create, update, archive, unarchive, picker) MUST be scoped to the current session's `userId` at the route boundary. No route handler MAY accept a `userId` parameter from request input; the session is the sole source of truth for the requesting user's identity. Cross-user reads and writes MUST be impossible to perform through the product surface.

#### Field rules

- **FR-004**: The `name` field MUST be a non-empty, whitespace-trimmed string between 1 and 80 characters (matching `Account.name`'s upper bound from feature 004 FR-004 for consistency). Validation MUST run at the boundary before any persistence. Names are not required to be unique per user.
- **FR-005**: The `kind` field MUST be a member of the closed enum `{INCOME, EXPENSE}`. A child category's `kind` MUST match its parent's `kind`; this rule MUST be enforced at the boundary on both create and edit. The kind of a category that has at least one child MUST NOT be changed; submitted edit payloads that attempt to do so MUST be rejected at the boundary with an actionable error.
- **FR-006**: The hierarchy MUST be at most a single level deep. A category whose `parentId` is null is a top-level category. A category whose `parentId` is non-null is a child category. A child category MUST NOT itself have children: any submitted payload that sets `parentId` to a category whose own `parentId` is non-null MUST be rejected at the boundary with an actionable error. A category's `parentId` MUST NOT equal its own id (no self-parent). The self-referential FK is retained at the data level so a future feature could lift the single-level restriction without a destructive migration.
- **FR-007**: The `color` field MUST be a member of a curated allow-list of color tokens (a small palette; the exact shape — e.g., a hex code, a named token, a palette index — is plan-level, but the validation rule is fixed: the value MUST belong to a closed, version-controlled set). Submitted values outside the allow-list MUST be rejected at the boundary with an actionable error. The allow-list MUST live in the codebase, not in a runtime lookup.
- **FR-008**: The `icon` field MUST be a member of a curated allow-list of icon identifiers (a small set of recognizable, visually-distinct icons covering common personal-finance categories — food, housing, transport, health, etc.). The exact icon library and the exact set of identifiers are plan-level, but the validation rule is fixed: the value MUST belong to a closed, version-controlled set. Submitted values outside the allow-list MUST be rejected at the boundary with an actionable error.
- **FR-009**: A category's `parentId`, when non-null, MUST reference an existing `Category` owned by the SAME `userId`, with a `kind` equal to this category's `kind`, AND with its own `parentId` equal to null (i.e., the parent is itself a top-level category). All three conditions MUST be enforced at the boundary on both create and edit. Failures collapse to a not-found-style envelope when the violation is a cross-user reference (FR-013), and to an actionable field-scoped error when the violation is structural (kind mismatch, would-be-grandchild, self-parent).

#### Archive

- **FR-010**: Archive is a soft state, not a delete. The system MUST expose an "Archive" action that sets `archivedAt` to the current server timestamp, and an "Unarchive" action that clears `archivedAt`. Both actions MUST be reversible an arbitrary number of times. The system MUST NOT expose any product surface that hard-deletes a category row. Archiving a parent category MUST NOT cascade-archive its children; children retain their own `archivedAt` state independently. The default list view MUST exclude rows where `archivedAt` is non-null; a "Show archived" toggle (default off) MUST make archived rows visible alongside active rows with a muted treatment and an "Archived" badge.
- **FR-011**: The default category picker (the primitive future features 006 and 008 will consume) MUST exclude rows where `archivedAt` is non-null. A future feature MAY introduce an "include archived" mode on the picker; the default in this feature MUST hide archived rows.

#### Seed

- **FR-012**: When a new user signs up, the system MUST seed a default category set in the same database transaction as the user-row creation. The seed MUST consist of exactly eleven categories: seven top-level EXPENSE ("Food", "Housing", "Transport", "Utilities", "Entertainment", "Health", "Other Expenses"), two top-level INCOME ("Salary", "Other Income"), and two child EXPENSE under "Food" ("Groceries", "Restaurants"). Each seeded category MUST have a stable default `color` and `icon` drawn from the curated allow-lists (FR-007, FR-008). If the seed write fails for any reason, the entire signup transaction MUST roll back; a signed-up user with zero categories MUST NEVER be a reachable state. The seeded categories are owned by the newly-created user and are independently editable, archivable, and (eventually) referenceable like any user-created category.

#### Error & validation contract

- **FR-013**: Operations on a category that does not belong to the session's user (read-by-id, update, archive, unarchive, or being referenced as a `parentId` from another category create/edit) MUST return a not-found-style error envelope (`{ error: { code, message } }` per the constitution). The system MUST NOT distinguish "this category belongs to another user" from "this category does not exist" in any user-visible response, log entry, or response-timing channel.
- **FR-014**: All Category API endpoints (server actions or route handlers) MUST validate input with Zod at the boundary before touching business logic or persistence, per constitution Principle III. Once validated, internal helpers MUST trust their typed inputs and MUST NOT re-validate.
- **FR-015**: All Category API endpoints MUST conform to the constitution's response envelope: success returns `{ data }`, failure returns `{ error: { code, message } }`. HTTP status reflects outcome (2xx success, 4xx client error, 5xx server error). Internal validation messages are surfaced as actionable field-scoped errors in the response, never as raw stack traces.

#### Surfaces

- **FR-016**: The feature MUST introduce a new top-level route at `/dashboard/categories` inside the authenticated shell (under the same shell chrome established by features 002 and 003). The route MUST appear in the sidebar's MANAGE group alongside Accounts. The Categories surface MUST use the existing brand vocabulary: the `<EmptyState>` primitive with its `illustration` and `preview` slots is the zero-state pattern (relevant when a user has archived every category and the default filter hides them), and the visual treatment matches the polish established by feature 005's branded-UI chore. No new top-level navigation grouping is introduced.
- **FR-017**: The create + edit form MUST be a side sheet rendered from the categories list, matching the pattern feature 004 established for Accounts. There is no separate per-category detail page in this feature. Creating a category opens the same sheet in "create" mode; clicking an existing row opens it in "edit" mode pre-populated with the current values.
- **FR-018**: The feature MUST introduce a single category picker primitive (a reusable control with a stable contract) intended for consumption by future features 006 (Transactions) and 008 (Budgets). The primitive MUST satisfy the rules in US4: it MUST list every non-archived category the session's user owns, MUST be grouped by `kind`, MUST visually express the parent/child hierarchy, MUST allow selecting either a parent or a child as an independent pick, and MUST return the picked category's stable identifier through a defined consumer interface. The exact rendering, the keyboard model, and the API shape are plan-level; the spec only locks the contract.
- **FR-019**: The categories list MUST be sorted deterministically and stably across reloads. Within each `kind`, top-level categories MUST be sorted alphabetically by name (case-insensitive, ascending), with children appearing immediately under their parent (also alphabetically by name within a parent's children). User-defined ordering and drag-to-reorder are OUT OF SCOPE for this feature; the deterministic alphabetical default matches feature 004 FR-012.

#### Quality

- **FR-020**: The categories list and the create/edit side sheet MUST be operable by keyboard alone. Form fields MUST have associated labels; error messages MUST be programmatically associated with their fields. Color MUST NOT be the sole carrier of category identity: `name` and `icon` MUST also be present on every rendered row so a user who cannot distinguish the color (low vision, color-blind, monochrome rendering) can still identify a category. The "Archived" badge MUST also be a text label (not a color-only indicator).
- **FR-021**: All new code MUST satisfy TypeScript strict mode with no use of `any` (constitution Principle II). All boundary schemas MUST be defined with Zod (constitution Principle II and III). No file outside the category module's boundary MUST perform direct validation on category input — the Zod schemas defined at the boundary are the single source of truth.
- **FR-022**: This feature MUST NOT introduce transactions, transfers, budgets, CSV import/export, charts, auto-categorization rules, per-category transaction counts, per-category spending totals, cross-user category sharing, or any per-category currency. Each is explicitly deferred to a later roadmap feature.

### Key Entities

- **Category** — A user's classification of income or expense (e.g., "Groceries", "Rent", "Salary"). One Category belongs to exactly one User. Identified by a stable unique identifier. Attributes:
  - `userId` — the owning user's identifier; the row is unreachable to any other user; cascade-deletes with the user.
  - `name` — a short user-visible label (1–80 chars after trim) such as "Groceries" or "Side Hustle". Not required to be unique per user.
  - `parentId` — nullable self-referential identifier. If null, the category is top-level. If non-null, it MUST reference another Category owned by the same user, with the same `kind`, and whose own `parentId` is null (single-level hierarchy; FR-006).
  - `kind` — one of a closed enum: `INCOME`, `EXPENSE`. Editable any time on a category with no children; immutable on a category with children (FR-005).
  - `color` — a token from a curated, version-controlled palette (FR-007). The exact representation (hex, named, indexed) is plan-level.
  - `icon` — an identifier from a curated, version-controlled set (FR-008). The exact icon library is plan-level.
  - `archivedAt` — nullable timestamp. Non-null means the category is archived (hidden from the default list and the default picker, retained for referential integrity). Reversible. Archiving a parent does NOT cascade to its children (FR-010).
  - `createdAt`, `updatedAt` — standard timestamps, stored UTC.
  - Relationship: every `Category` belongs to exactly one `User`. A `User` may own zero or many `Categories` (though a brand-new user always starts with the eleven seeded categories per FR-012). A `Category` MAY reference exactly one parent `Category` (also owned by the same user, also matching `kind`, also a top-level category). There is no shared-category model; a `Category` is never owned by two users.
- **(Future-only references — NOT created in this feature)** — `Transaction` (feature 006) will carry an optional `categoryId` referencing `Category.id`; `Budget` (feature 008) will carry a `categoryId` referencing `Category.id`. This feature does NOT add `Transaction` or `Budget` and MUST NOT presume their shape beyond "they will reference this category by id, and the category's `userId` will match the consumer's `userId` because of the data-scoping convention." The category picker primitive defined in FR-018 is the canonical handoff surface for those features.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A newly-signed-up user navigating to `/dashboard/categories` for the first time sees the eleven seeded categories within the first 5 seconds of landing on the page, in 100% of attempts. No empty state is rendered for a freshly-seeded user.
- **SC-002**: A user can create a new category (name + kind + color + icon, plus optional parent) in under 30 seconds of interaction time from opening the create CTA to seeing the new row in the list.
- **SC-003**: A SECOND newly-signed-up user visiting `/dashboard/categories` sees only THEIR OWN eleven seeded categories — none of the first user's categories appear in the list, the parent picker, or any picker context — in 100% of attempts. This is the data-scoping convention exercised for the second domain entity (after Accounts in feature 004).
- **SC-004**: Editing any editable field on an existing category (name, kind on a no-children category, color, icon, parent) reflects the change in the list row within one normal interaction (submit → list updates) in 100% of attempts.
- **SC-005**: Archiving a category hides the row from the default list in 100% of attempts. Toggling "Show archived" reveals the archived row with a visible "Archived" badge in 100% of attempts. Unarchiving the same category restores it to the default list in 100% of attempts. Archiving a parent does NOT remove its children from the default list in 100% of attempts.
- **SC-006**: Submitting an invalid payload (blank name, name beyond 80 chars, kind missing or out of enum, color or icon outside the curated allow-list, `parentId` not found, `parentId` cross-user, `parentId` kind-mismatched, `parentId` would-be-grandchild, `parentId` equal to self) is rejected at the Zod boundary and never reaches persistence in 100% of attempts. The list state on the page is unchanged after such a rejection.
- **SC-007**: Cross-user access attempts (reading, updating, archiving, unarchiving a category belonging to a different user, OR creating/editing a category whose `parentId` references a category belonging to a different user) MUST return a not-found-style error envelope in 100% of attempts. No user-visible distinction between "not yours" and "does not exist" is exposed in response body, response headers, or response timing.
- **SC-008**: A signup failure during seeding (any reason: DB error, constraint violation, transient I/O failure) MUST result in a fully-rolled-back signup — no `User` row, no `Category` rows — in 100% of attempts. A user with no categories MUST NEVER be observable post-signup.
- **SC-009**: The category picker primitive, when mounted in two different consumer contexts on the same page or in two separate pages, renders the same list, responds to selection the same way, and returns the same kind of identifier in 100% of attempts. There is exactly one picker implementation, not two parallel ones.
- **SC-010**: The same parent/child shape locked by this feature is consumable by feature 006 (Transactions) and feature 008 (Budgets) WITHOUT any subsequent migration to the `Category` model. The schema lands once, here.
- **SC-011**: All category management interactions (list navigation, opening the create CTA, opening an edit sheet on a focused row, filling and submitting the form, toggling "Show archived", activating Archive / Unarchive) are operable end-to-end with the keyboard alone in 100% of attempts. Color is never the sole carrier of category identity in the rendered list.
- **SC-012**: A type-check of all new code passes with strict mode enabled and zero uses of `any` in 100% of runs.
- **SC-013**: Every existing Vitest unit test and every existing Playwright e2e test on `main` continues to pass after this feature lands. New unit tests cover at minimum the parent-validation rules (FR-006, FR-009), the kind-mismatch and would-be-grandchild rules, and the seed contents.
- **SC-014**: The categories list, the create/edit side sheet, the empty state (rendered only when the user has archived all categories and the default filter hides them), and the category picker primitive all use the brand vocabulary established by feature 005's polish (the same `<EmptyState>` primitive with its illustration and preview slots, the same sidebar grouping, the same shell chrome) in 100% of rendered states. No bespoke one-off styling is introduced.

## Assumptions

- Feature 003's auth boundary is in place: every `/dashboard/*` route requires a session; the session is the canonical source of the requesting user's identity. This feature builds on that boundary without re-implementing it.
- Feature 004's `Account` model and its data-scoping pattern (cascade FK to `User`, indexed by `userId`, queries always scoped to session `userId`, cross-user collapse to not-found, archive-not-delete, side-sheet edit UX) are on `main`. This feature reuses the same conventions for the second domain entity. There is no schema-level coupling between `Account` and `Category` in this feature; they are independent sibling tables both rooted at `User`.
- Feature 005's branded UI (the `<EmptyState>` primitive with `illustration` and `preview` slots, the `<Money>` primitive though not used here, the `<ShellFooter>`, the MANAGE / TRACK sidebar grouping) is on `main` and is the visual vocabulary this feature uses. No new primitives are introduced beyond the category-specific picker.
- The seed write (FR-012) happens in the SAME database transaction as the user-row creation that feature 003 owns. The plan stage will decide exactly where in the auth code path the seed insert is wired and whether it lives in the category module's helpers or in the auth module's signup helper, but it MUST be one transaction, not two sequential ones.
- The exact color palette and icon set (FR-007, FR-008) are plan-level decisions. The spec locks the shape (curated, closed, version-controlled, validated at the boundary) but not the specific tokens. The plan stage MAY choose a small custom palette and the lucide-react icon set already in the project, or any equivalent; whichever it chooses, the same allow-list MUST cover both the user's create/edit form (the picker UX) and the seed defaults (FR-012).
- The category picker primitive (FR-018) is introduced in this feature for downstream consumption but is not itself mounted in any production consumer in this feature. The plan stage MAY include a small isolated demo surface for testing purposes; the spec only locks the contract.
- No `/dashboard/categories` placeholder currently exists in the routes tree (the dashboard currently has `accounts`, `budgets`, `settings`, and `transactions` placeholders from feature 002 / the 005 chore). This feature introduces the route fresh and wires it into the sidebar's MANAGE group.
- This feature does NOT introduce `lib/money/` work — categories are not monetary entities. They have no `currency` field and no `amount` field. Categories classify money; they do not hold money. (Constitution Principle I is therefore not in scope for this feature beyond the existing `lib/money/` from feature 004 remaining untouched.)
- The Playwright E2E surface for this feature is plan-level. The spec's required automated coverage is the unit-level validation suite for FR-005 / FR-006 / FR-009 (parent rules, kind rules, single-level depth) and the seed-correctness test for FR-012.

## Out of Scope

- **Transactions** — adding, editing, deleting, or listing transactions; transactions referencing categories. Lands in feature 006.
- **Transfers** — atomic two-ledger-entry transfers between accounts. Lands in feature 006. (Transfers are inherently uncategorized, but their existence is locked there, not here.)
- **Budgets** — capping spending by category against an account. Lands in feature 008. The category schema this feature ships is the one feature 008 will reference.
- **Auto-categorization rules** — pattern-match a payee or description to a category and apply it automatically. Lands in feature 012. Categories must exist before rules can target them; this feature unblocks 012 without implementing any of it.
- **Per-category transaction counts / spending totals** — aggregate widgets in the categories list. Lands with feature 016 (Reports). The list in this feature is strictly row-level: name, color, icon, kind, parent, archive state. No counts, no totals.
- **CSV import that maps categories from a bank file** — feature 010. The category column-mapping UX is owned there, not here.
- **CSV export of categories** — feature 014 ships the export format. Per-category lists are not separately exportable.
- **Drag-to-reorder, user-defined ordering** — sort is deterministic-alphabetical-by-name within each kind. User-defined ordering may land in a future polish feature; it does not land here.
- **Hard delete of categories** — only archive is exposed. A future feature MAY introduce hard delete with safeguards (e.g., only categories never referenced by any transaction); this feature does not.
- **Per-category notes, descriptions, or long-form metadata** — a category is a small entity with a name, a kind, a color, an icon, and a parent. Additional metadata is a future polish item, not in this feature.
- **Custom color picker / custom icon picker that accept user-supplied values** — colors and icons come from a curated, closed allow-list (FR-007, FR-008). A free-form color or icon picker is out of scope; users pick from the curated set or they don't pick at all.
- **Cross-user category sharing or merging** — Abacus has no shared workspaces per feature 003. A category belongs to exactly one user and is never visible to another user.
- **Category-level currency** — categories do not have a currency. Transactions inherit currency from their `Account`, not from their `Category`. There is no multi-currency consideration in this feature.
- **Importing categories from another finance app** — out of scope. Users start with the eleven seeded defaults and customize from there.
- **A "default category" or "uncategorized" placeholder for transactions without a category** — out of scope. How feature 006 handles uncategorized transactions (a sentinel category vs. a nullable `categoryId`) is a feature-006 decision; this feature does not pre-empt it.
- **Multi-level (depth > 1) hierarchy** — the data model uses a self-referential FK, but FR-006 restricts depth to one level. Lifting that restriction is a future feature if user feedback justifies it; this feature does not.
- **Per-category icons or colors customized per user beyond picking from the palette** — users pick from the curated palette and icon set. They cannot upload their own icon or pick an arbitrary RGB color.
